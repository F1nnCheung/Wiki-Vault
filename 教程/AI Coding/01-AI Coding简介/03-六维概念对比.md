# 1.3 六维概念对比

> 上一章逐一解释了每个概念。这一章把它们放在一起对比，帮你建立层次感和协作关系——这是后续选择和使用工具的认知基础。

---

## 概述

Prompt、命令、Rule、Skills、MCP、Agent——这六个概念本质上是 **从不同维度、不同层级解决同一个问题：如何让 AI 更准确、更自主地完成任务**。

理解它们的差异和协作关系，是高效使用任何 AI Coding 工具的前提。

---

## 六维对比总表

| 维度 | Prompt | 命令 | Rule | Skills | MCP | Agent |
|---|---|---|---|---|---|---|
| **触发方式** | 每次手动输入 | 手动 `/` 触发 | **自动加载** | **自动发现调用** | AI 按需调用 | **自主决策** |
| **持久性** | 一次性 | 一次性 | 跨会话持久 | 跨会话持久 | 跨会话持久 | 跨会话持久 |
| **内容形态** | 自然语言 | 快捷指令模板 | 项目规范文档 | Markdown 方法论包 | 外部工具接口 | 完整决策系统 |
| **定位** | 告诉 AI **做什么** | 快速执行**常见操作** | 告诉 AI **边界在哪** | 教 AI **怎么思考** | 给 AI **工具** | AI **自主行动** |
| **层级** | L0：基础交互 | L1：快捷操作 | L1：持久约束 | L2：方法论封装 | L2：工具扩展 | L3-L5：自主智能 |
| **类比** | 口述需求 | 快捷键 | 公司规章制度 | 员工培训手册 | 办公设备 | 自主员工 |

## 逐个深入对比

### 纵比一：Prompt vs Rule

两者都涉及「告诉 AI 做什么」，但差异巨大：

| | Prompt | Rule |
|---|---|---|
| 生效范围 | 仅当前会话 | 所有会话自动加载 |
| 内容 | 本次具体需求 | 项目长期规范 |
| 修改频率 | 每次不同 | 项目稳定后很少改 |
| 典型内容 | "帮我实现用户登录" | "使用 TypeScript 严格模式，所有 API 放 src/api/" |

> 💡 **实践技巧**：好的 Rule 让你每次 Prompt 可以更短。因为你不需要在每次请求里重复解释技术栈和规范。

### 纵比二：命令 vs Skills

两者都封装了可复用操作，但触发方式和复杂度完全不同：

| | 命令 | Skills |
|---|---|---|
| 触发 | 手动输入 `/命令名` | AI 自动判断时机调用 |
| 适用场景 | 简单、重复操作 | 复杂、多步骤流程 |
| 内容 | 一段提示模板 | 完整方法论（流程文档+脚本+模板） |
| 举例 | `/compact` 压缩上下文 | Superpowers 全流程增强（14 个方法论） |

> 💡 **选择原则**：如果你每次都要记得在特定时机手动触发，说明它应该是 Skill 而不是命令。Skill 让 AI 在合适的时机自动执行。

### 纵比三：Skills vs MCP

这可能是最容易被混淆的两个概念：

| | Skills | MCP |
|---|---|---|
| 解决什么问题 | 教 AI **怎么思考** | 给 AI **工具** |
| 本质 | Markdown 文档（方法论） | 外部服务接口（工具） |
| 举例 | "做前端设计时分五步走……" | 操作浏览器、查询数据库 |
| 安装方式 | 放到 skills 目录 | `claude mcp add` 命令 |

> 💡 **记忆口诀**：Skills = 大脑（思考方式），MCP = 手脚（执行工具）。

### 纵比四：Rule vs Agent

Rule 定义 Agent 的边界，Agent 在边界内自主行动：

```
Rule（约束层）
  ↓ 定义「什么能做、什么不能做」
Agent（决策层）
  ↓ 在约束内自主决定「怎么做」
```

没有 Rule 的 Agent 是危险的（可能做任何事），没有 Agent 的 Rule 是没有执行力的（只有规范没有行动）。

## 六者协作关系

### 协作架构图

```
Rule（约束层）
  ↓ 定义边界和规范，每次会话自动注入
Prompt（指令层）
  ↓ 描述本次具体需求
Agent（决策层）
  ↓ 理解需求 → 制定计划 → 自主执行
  ├── 调用 Skills（方法论层）
  │     ↓ 按流程思考、按模板输出
  ├── 调用 MCP（工具层）
  │     ↓ 操作浏览器/数据库/GitHub
  └── 响应 /命令（快捷层）
        ↓ 执行常规操作
```

### 协作实例：实现一个用户登录功能

看看六者如何在实际任务中协作：

```
1. Rule 自动加载：
   "本项目使用 Next.js 14 + TypeScript + Prisma，
    API 放 src/app/api/，需要 JWT 认证"

2. 你输入 Prompt：
   "实现用户登录功能，包括邮箱密码登录和 JWT token 生成"

3. Agent 自主规划：
   ① 查看现有数据库 schema
   ② 创建 API 路由
   ③ 实现登录逻辑
   ④ 写单元测试

4. Agent 调用 Skills（前端设计 Skill）：
   "按照 UI 设计规范生成登录页面组件"

5. Agent 调用 MCP（数据库 MCP）：
   "查询 users 表结构，确认字段名称"

6. 你手动使用命令：
   /diff  → 查看所有改动
   /cost  → 查看本次 Token 消耗
```

## 能力层级模型

把六者按照「人类参与度」和「AI 自主度」放在一条谱系上：

```
人类主导 ←——————————————————————————→ AI 主导

Prompt    命令    Rule    Skills    MCP    Agent
  │        │       │        │        │       │
  │        │       │        │        │       └─ 几乎全自主
  │        │       │        │        └─ 工具自主调用
  │        │       │        └─ 方法自动匹配
  │        │       └─ 约束自动生效
  │        └─ 手动快捷触发
  └─ 每次手动输入
```

## 一句话速记

| 概念 | 一句话 |
|---|---|
| **Prompt** | 告诉 AI「这次要做什么」 |
| **命令** | 快速触发「常见操作」 |
| **Rule** | 告诉 AI「在这个项目里怎么做事」 |
| **Skills** | 教 AI「遇到某类问题怎么思考」 |
| **MCP** | 给 AI「连接外部世界的手脚」 |
| **Agent** | 以上所有要素整合后的「自主行动者」 |

## 常见误区

### 误区一：把 Prompt 当 Rule 用

❌ 每次新会话都复制粘贴一大段技术栈说明
✅ 把技术栈和规范写进 CLAUDE.md，Prompt 只描述本次需求

### 误区二：把所有操作都做成 Skill

❌ 把「格式化代码」做成 Skill
✅ 格式化是简单操作，用命令（`/format`）即可。Skill 留给复杂流程

### 误区三：装了 MCP 就不用 Skills

❌ "我装了 Playwright MCP，AI 就能做好前端测试了"
✅ MCP 只给了 AI 操作浏览器的能力，Skills 才告诉它「怎么做好测试」

### 误区四：Rule 写得太长

❌ 写了 2000 行的 CLAUDE.md，每次会话都消耗大量 Token
✅ Rule 控制在 200 行以内，聚焦核心规范。详细文档放外部引用

---

## 本章小结

1. **六者不是替代关系，是分层协作关系**——每一层解决一个特定维度的问题
2. **Rule 是基础**——先写好 Rule，其他层才能高效运作
3. **Skills 和 MCP 是能力放大器**——Skills 扩思维，MCP 扩工具
4. **Agent 是终极形态**——当六层齐全，Agent 能实现接近自主的开发流程
5. **理解层次关系 > 记住定义**——知道「什么时候用什么」比「每个是什么」更重要
