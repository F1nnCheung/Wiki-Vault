# 1.2 核心概念名词解释

> AI Coding 领域充满了各种新名词：Vibe Coding、Agent、MCP、Skills……每个概念代表一种思路。本章帮你一次性理清这些概念的含义、关系和层次。

---

## 概述

在深入学习工具之前，必须先建立概念地图。这些概念从底层到顶层构成了一条完整的链条：

```
Prompt（指令）→ Rule（规范）→ Skills（方法论）→ MCP（工具）→ Agent（自主行动）
```

每一个层次都解决一个特定问题。理解它们的定位和关系，是后续高效使用任何 AI Coding 工具的前提。

---

## Vibe Coding（氛围编程）

> 2025 年初由 **Andrej Karpathy**（前 OpenAI 联合创始人、前 Tesla AI 总监）提出。

### 定义

用自然语言描述想要的效果，让 AI 生成代码——开发者「凭感觉」（vibe）编程，关注「做什么」而非「怎么做」。

### 核心理念

- **不逐行审查代码**，而是描述期望效果
- 让 AI 反复迭代直到满意
- 降低编程门槛，让非技术人员也能开发
- 快速原型验证，加速从想法到产品的过程

### 实操示例

```
❌ 传统编程：
"我要写一个函数，参数是用户ID，先查数据库获取用户信息，
然后格式化日期字段，再计算用户等级，返回JSON响应..."

✅ Vibe Coding：
"给我做一个用户信息接口，包含基本信息和等级计算，
日期格式用 YYYY-MM-DD"
```

### ⚠️ 风险警示

Vibe Coding 不是银弹，它有三个严重风险：

1. **「技术次贷危机」**：AI 垃圾代码堆积——生成成本趋近于零，但理解和维护成本指数上升
2. **「死亡螺旋」**：错误代码喂给 AI → AI 反刍出新错误 → 人类逐渐丧失对代码库的控制
3. **「技能退化」**：过度依赖 AI 可能导致开发者丧失底层理解能力

> 💡 **如何规避**：Vibe Coding 最适用于原型验证和探索阶段。一旦代码要进入生产环境，必须回归到结构化开发流程——写测试、做 Code Review、保持 Spec 驱动。

### 🔧 实战工具链

一个典型的 Vibe Coding 全流程（以微信小程序为例）：

```
Figma（原型设计）→ Trae（AI 编码）→ HBuilderX（项目框架）→ 微信开发者工具（运行）
```

- 用自然语言在 Figma 描述原型（如「创建一个宝妈馄饨的微信小程序」）
- 将原型图拖入 Trae，prompt：「参考原型图，一比一复刻开发前端页面」
- 配合 HBuilderX（UniApp + UniCloud）搭建多端项目框架
- 在微信开发者工具中运行并发布

> 📖 详见：[[Wiki/wiki/concepts/vibe-coding|Vibe Coding 概念页]] — 含完整实践案例和工具链详解

## Agentic Engineering（智能体工程）

> 2026 年初由 Karpathy 提出，是 Vibe Coding 的进化形态。

### 定义

多个 AI 智能体自主完成 **规划→编写→测试→审查** 全流程。人类从「驾驶员」退后为「架构师」和「监督者」。

### Vibe Coding vs Agentic Engineering

| 维度 | Vibe Coding | Agentic Engineering |
|---|---|---|
| 人类角色 | 需求描述者 | 架构师 / 监督者 |
| AI 角色 | 单一代码生成器 | 多 Agent 协同团队 |
| 流程 | 人→AI→人检查 | AI 自主规划→执行→验证 |
| 产出质量 | 不确定，全靠描述质量 | 有审查闭环保证 |
| 适合阶段 | 原型探索 | 生产交付 |

### 角色分工

```
Planner（规划） → Coder（编码） → Tester（测试） → Reviewer（审查）
                                                          ↓
                                              Human 架构师（定义边界、验收）
```

## ID Coding（意图编程）—— 终极形态

### 定义

人类表达意图 → 直接转化为软件服务。编程语言本身可能消失，取而代之的是意图表达和架构设计能力。

### 对未来开发者的影响

| 变化方向 | 技能 |
|---|---|
| ↓ 贬值 | 写代码（语法、API 记忆、样板代码） |
| ↑ 升值 | 读代码、定架构、系统设计、需求理解 |
| → 持续重要 | 代码审查（质量把控的最后防线） |

> 💡 **启示**：与其花时间背 API 文档，不如深入理解系统架构和需求分析。这些才是 AI 难以替代的能力。

## MCP（Model Context Protocol）

> AI 连接外部工具和数据源的开放标准协议。类比 USB-C——为 AI 接入外部系统提供标准化方式。

### 为什么需要 MCP

默认情况下，Claude Code 只能做三件事：读文件、写文件、对话。这相当于一个只能「想」但不能「动手」的大脑。

MCP 让它能：

- 🖥️ 操作浏览器（Playwright MCP）
- 📦 管理仓库（GitHub MCP）
- 📚 查阅最新文档（Context7，防止 AI 幻觉）
- 🗄️ 连接数据库（PostgreSQL/SQLite MCP）
- 🔧 桌面控制（Desktop Commander）

### 工作原理

```
AI ←→ MCP 协议 ←→ MCP Server ←→ 外部系统
                        ├── Playwright Server → 浏览器
                        ├── GitHub Server → GitHub API
                        ├── Database Server → PostgreSQL
                        └── ...
```

> ⚠️ **社区共识**：不装 MCP 的 Claude Code 只发挥了约三成功力。

## Skills（技能系统）

### 定义

给 AI 安装的可复用「能力包」——本质是特定目录下的 Markdown 文档，AI 自动发现并在适当时机按需调用。

### Skills vs 其他概念

| 维度 | Skills | Slash 命令 | MCP |
|---|---|---|---|
| 触发方式 | **自动发现、智能调用** | 手动输入 `/命令` | AI 按需调用 |
| 内容 | 完整方法论（流程+脚本+模板） | 提示文本模板 | 工具接口 |
| 定位 | 教 AI **怎么想** | 快速指令 | 给 AI **工具** |
| 安装位置 | `~/.claude/skills/` | `.claude/commands/` | MCP 配置 |

### Skill 文件结构示例

```
skill-name/
├── SKILL.md       # 必需：技能说明和元数据
├── scripts/       # 可选：辅助脚本
├── templates/     # 可选：文档模板
└── resources/     # 可选：参考文件
```

> 💡 **理解要点**：Skill 是最被低估的概念。一个好的 Skill 等于把资深开发者的工作方法论「编码」进了 AI 的执行流程里。

## Agent（智能体）

### 定义

能自主感知环境、制定计划、执行多步任务并自我修正的 AI 系统。

### 能力五层次

```
L1: 对话 AI      → 一问一答，无记忆
L2: 代码补全      → 实时建议，单次操作
L3: 任务执行      → 多步操作（读文件→写代码→跑测试）
L4: 自主 Agent    → 规划→执行→验证→修正 全闭环
L5: 多 Agent 协同 → 多个 Agent 分工协作，人类监督
```

### 核心五要素

```
感知（读取环境信息）
  ↓
规划（分解目标为子任务）
  ↓
执行（调用工具完成子任务）
  ↓
记忆（短期上下文 + 长期持久化）
  ↓
反思（自我评估 + 修正策略）
```

> 💡 **判断标准**：一个系统能不能叫 Agent，关键看它有没有「规划→执行→反思」的自主闭环。没有自我修正能力的只是「自动化脚本」，不是 Agent。

## Prompt（提示词）

### 定义

你向 AI 描述需求的自然语言指令。是所有 AI 交互的起点——门槛最低，但天花板也最高。

### Prompt 的层次

```
🟡 一句话需求：
   "帮我写个登录页"
   → AI 自由发挥，结果不可控

🟠 结构化需求：
   "用 React + Tailwind，包含邮箱/密码输入、
    表单验证、错误提示、提交按钮"
   → 更可控，但仍靠 AI 自行理解

🟢 Spec-Driven：
   "参照 spec/auth.md 接口定义，
    实现 src/pages/Login.tsx"
   → 精确约束，AI 按规格执行
```

### 为什么 Prompt 仍然重要

- Skills、MCP、Agent 本质都是「封装好的高级 Prompt」
- 好的 Prompt 能减少返工、降低 Token 消耗
- 即使有了 Skill，最后的细节修正仍然靠 Prompt
- Prompt 是唯一不需要任何配置就能用的能力——在紧急情况下最可靠

## 命令（Commands / Slash Commands）

### 定义

AI Coding 工具中手动触发的快捷指令，以 `/` 开头。

### 常用命令分类

**会话管理**：
- `/clear` — 重置会话
- `/compact` — 压缩上下文（保留主线）
- `/resume` — 恢复历史会话

**上下文控制**：
- `/context` — 查看上下文占用
- `/cost` — 查看费用
- `/model` — 切换模型

**任务执行**：
- `/init` — 初始化项目记忆文件
- `/plan` — 进入规划模式
- `/todos` — 任务管理

**代码操作**：
- `/diff` — 查看改动
- `/rewind` — 回退操作
- `/simplify` — 代码评审

### 命令 vs Skills 的关键区别

> **命令是手动触发，Skills 是自动发现。** 命令适合简单重复操作（如一键格式化），Skills 适合复杂流程（如完整的前端设计方法论）。

## Rule（规则 / 项目记忆）

### 定义

持久化的指令文件，告诉 AI「在这个项目里应该怎么做」。不同于一次性 Prompt，Rule 在**每次会话自动加载**，形成 AI 的「长期项目记忆」。

### 各工具的 Rule 体系

| 工具 | Rule 文件 | 作用域 | 格式 |
|---|---|---|---|
| Claude Code | `CLAUDE.md` | 项目级 / 用户级 | Markdown |
| Cursor | `.cursorrules` | 项目级 | Markdown |
| Trae | `.trae/rules.md` | 项目级 | Markdown |
| Hermes Agent | `AGENTS.md` + `SOUL.md` | 项目级 + 用户级 | Markdown |

### Rule 里写什么

以 `CLAUDE.md` 为例：

```markdown
# 项目规范
- 技术栈：Next.js 14 + TypeScript + Prisma + Tailwind
- 使用 Server Components 优先
- 所有 API 路由统一放 src/app/api/
- 测试用 Vitest，覆盖率 >80%
- 禁止直接修改 .env 文件
```

### Rule 的核心价值

不用每次新会话都重复解释项目背景、技术栈、编码规范。Rule 文件越完善，AI 的「理解偏差」越少，返工越少。

> 💡 **实践建议**：装完任何 AI Coding 工具后，第一件事就是写好 Rule 文件。这 5 分钟投入能节省后续几十个小时的反复纠正。

## 概念关系速查

```
Prompt（指令）  ← 每次手动输入，描述「这次要做什么」
Rule（规范）   ← 自动加载，定义「在这个项目里怎么做」
命令（快捷）   ← 手动触发，执行「常见操作」
Skills（方法） ← 自动发现，教 AI「遇到某类问题怎么思考」
MCP（工具）   ← 按需调用，给 AI「连接外部世界的手脚」
Agent（自主）  ← 整合以上全部，成为「自主行动者」
```

---

## 本章小结

1. **Vibe Coding → Agentic Engineering → ID Coding** 是一个从「人主导」到「AI 主导」的演进谱系
2. **MCP** 给 AI 工具，**Skills** 给 AI 方法论，**Rule** 给 AI 边界，**Prompt** 给 AI 指令
3. **Agent** 是以上所有要素的终极整合——能自主规划、执行、反思的 AI 系统
4. 这些概念不是相互替代，而是**分层协作**——理解层次关系比记住定义更重要
