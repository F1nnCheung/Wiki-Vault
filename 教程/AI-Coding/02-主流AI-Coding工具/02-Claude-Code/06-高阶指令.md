# 2.2.6 Claude Code 高阶指令与技巧

> 掌握了基本命令和进阶功能（Skill/MCP）后，这些高阶指令和技巧能让你从「熟练用户」跃升为「高手」。关键是：知道在什么场景用哪个。

---

## 概述

以下命令不是日常必用，但在特定场景下能省去大量时间。按使用频率从高到低排列。

---

## 高阶命令速查

| 命令 | 作用 | 典型场景 |
|---|---|---|
| `/btw` | 不中断主任务快速提问 | 写代码时突然想到一个相关问题 |
| `/fast` | 极速模式，简化回复 | 简单操作不想看长篇大论时 |
| `/plan` | 只读规划模式 | 架构设计、方案讨论（安全第一） |
| `/todos` | 跨会话任务管理 | 大型任务拆分、进度追踪 |
| `/simplify` | 代码评审 | 提交前检查代码质量 |
| `/batch` | 大规模拆分改造 | 重构大型文件、批量迁移 |
| `/loop` | 持续观察/轮询 | 监控部署状态、观察构建进度 |

---

## 逐个详解

### `/btw` — 不打断主线的旁路提问

**场景**：你正在让 AI 实现一个复杂功能，突然想到一个相关问题，但不想打断当前任务的上下文。

```
# 使用
/btw 这个项目的认证用的什么方案？

# AI 会简短回答，然后继续之前的主任务
```

**为什么重要**：普通的提问会「污染」对话历史。`/btw` 让你在主线之外快速获取信息，不影响主任务的质量。

### `/fast` — 极速模式

**场景**：你只需要 AI 做一个简单的操作，不想看它长篇大论地解释。

```
/fast 把这个变量的类型改成 string
```

AI 会用最简短的方式响应，跳过「分析」「建议」「总结」等常规环节。

### `/plan` — 规划模式

**场景**：进入只讨论不修改的安全模式，适合架构设计和方案讨论。

```
/plan
我们要重构用户认证模块，当前是 JWT 方案，想加入 OAuth 2.0 支持。
先帮我分析现有代码结构，提出重构方案。
```

在规划模式下，AI 不会修改任何文件——可以放心讨论。

### `/todos` — 跨会话任务管理

**场景**：大任务需要多次会话才能完成。

```
/todos
1. 重构 User model 添加 OAuth 字段
2. 创建 OAuth service
3. 更新 auth middleware
4. 添加 OAuth callback 路由
5. 写测试
```

任务列表跨会话持久，下次启动时可以继续。

### `/simplify` — 代码评审

**场景**：提交代码前，用 AI 做一次全面评审。

```
/simplify
```

AI 会检查：
- 代码重复和冗余
- 可简化或合并的逻辑
- 命名和结构优化机会
- 潜在的 Bug 和边界情况

> 💡 建议配合 `/diff` 使用：先 `/diff` 看改了什么，再 `/simplify` 检查质量。

### `/batch` — 大规模拆分改造

**场景**：重构大型文件、批量迁移代码。

```
/batch 把 src/utils/ 下所有文件的 class 改成函数组件
```

AI 会逐个处理，每完成一个报告状态。

### `/loop` — 持续观察

**场景**：需要 AI 持续监控某个状态。

```
/loop 每 5 分钟检查一次部署状态，如果失败通知我
```

适合监控部署、构建、测试等需要等待的任务。

## 会话管理进阶技巧

### 何时用 compact vs clear vs resume

```
场景判断：
├── 上下文满了，但任务没完成 → /compact
├── 上下文乱了，AI 开始幻觉 → /clear
├── 会话意外中断，想继续 → claude -c（或 /resume）
└── 想回顾之前的某个会话 → /resume 选择历史会话
```

### 分段工作法

对于大型任务，不要在一个会话里死磕：

```
会话 1：设计和规划（/plan + 讨论）
  ↓ /compact 收尾
会话 2：核心实现
  ↓ /compact 收尾
会话 3：测试和修复
  ↓ /diff + /simplify
会话 4：代码审查和提交
```

每个会话聚焦一个阶段，避免上下文膨胀。

## 效率优化技巧

### 1. 简单任务用便宜模型

```bash
/model       # 查看可用模型
/model Haiku    # 切换到便宜模型
```

| 任务类型 | 推荐模型 |
|---|---|
| 代码补全、简单修改 | Haiku / DeepSeek |
| 复杂逻辑、架构设计 | Opus / Sonnet |
| Bug 调试 | Sonnet（性价比最佳） |

### 2. 用 `/effort` 控制推理深度

```bash
/effort low     # 简单操作：改个变量名、格式化
/effort medium  # 常规开发：实现功能、写测试
/effort high    # 复杂任务：架构设计、深度调试
```

### 3. 善用 `--print` 脚本集成

```bash
# 在 CI/CD 中集成 Claude Code
claude --print "检查 src/ 下所有文件是否有 TypeScript 类型错误" > report.md
```

## 常见问题与最佳实践

### Q: Claude Code 开始胡说八道了怎么办？

```
1. /context 检查上下文占用
2. 如果 >80%，立即 /compact
3. 如果 compact 没用，/clear 重来
4. 在 CLAUDE.md 中补充相关信息，避免下次重复解释
```

### Q: 我的 CLAUDE.md 写了 2000 行，有问题吗？

❌ 有问题。CLAUDE.md 应该控制在 200 行以内。

优化策略：
- 核心规范留 CLAUDE.md（技术栈、命名约定、禁止事项）
- 详细文档放外部文件，用 CLAUDE.md 引用
- 项目架构说明用 Repomix Skill 替代

### Q: AI 反复犯同一个错误怎么办？

把规则写入 CLAUDE.md：
```markdown
- ⚠️ 不要使用 any 类型，所有函数必须有完整类型注解
- ⚠️ API 路由使用 Next.js Route Handler，不要用 Express
```

### Q: 想在公司推广 Claude Code，有什么建议？

1. 先写团队级 `CLAUDE.md`（共享编码规范）
2. 让每人维护自己的 `~/.claude/CLAUDE.md`（个人偏好）
3. 按项目配置 `.mcp.json`（项目级 MCP）
4. 先从非核心模块开始使用，验证效果后推广

## 本章小结

1. `/btw` 和 `/fast` 是效率神器——短提问不打断主线
2. `/plan` 先讨论再执行，安全性大幅提升
3. 分段工作法：每个会话聚焦一个阶段
4. 简单任务用便宜模型，复杂任务用贵模型
5. CLAUDE.md 控制在 200 行以内
