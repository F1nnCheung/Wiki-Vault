## 第一部分：知识库技术总览

### 什么是知识库技术

知识库技术解决一个核心问题：**如何让计算机理解、组织和检索人类的知识**。它从「Ctrl+F 关键词搜索」一路演进到「AI 沿着关系链自主推理」，经历了三代变革。

### 三代演进

| 阶段 | 代表技术 | 核心能力 | 局限 |
|------|---------|---------|------|
| **第一代：关键词搜索** | Elasticsearch、全文索引 | 找到包含某个词的文档 | 搜"轿车"找不到"小汽车" |
| **第二代：语义检索** | 向量数据库 + Embedding | 理解语义相似性 | 只能找相似内容，不懂关系 |
| **第三代：知识理解** | GraphRAG、Agentic RAG、LLM Wiki | 理解实体关系、自主推理、知识积累 | 工程复杂度高 |

### 本教程覆盖的核心知识域

```
知识库技术
├── RAG 架构（Classic / Graph / Agentic）
├── RAG 优化管线（20 种方法）
├── 知识图谱（三元组 / 本体 / GraphRAG）
├── 代码知识图谱（GitNexus / Graphify）
├── LLM Wiki 范式（编译器类比）
├── 向量数据库 & 嵌入模型
└── 工具生态（Milvus / ChromaDB / 开源实现）
```

## 第二部分：RAG 三大架构

RAG（Retrieval-Augmented Generation，检索增强生成）的核心思路很简单：**先把相关资料找出来，再让大模型基于这些资料回答**。但发展至今已分化出三种截然不同的架构。

三个动词概括本质差异：
- **Classic RAG**：检索（Retrieves）— 找资料
- **Graph RAG**：连接（Connects）— 找关系
- **Agentic RAG**：推理（Reasons）— 决定下一步

### 2.1 Classic RAG — 检索：找资料

```
文档 → 切块（Chunk）→ Embedding → 向量数据库
查询时：用户问题 → Embedding → Top K 检索 → 拼入上下文 → LLM 生成答案
```

**适用场景**：FAQ 问答、政策查询、产品手册、客服知识库——任何「答案就在某个段落里」的问题。

**优势**：简单、快、成本低、工具链成熟、行为可预测。

**局限**：不擅长跨文档、跨实体的多跳问题。例如「A 的经理的经理是谁？」——答案不在一个段落，而在关系链中。

> Classic RAG 解决的是「**资料在哪里**」的问题。

### 2.2 Graph RAG — 连接：找关系

在文本检索之外增加一层**知识图谱**结构。实体 = 节点（人/产品/系统），关系 = 边（汇报/使用/依赖）。

```
文本线：切块 → Embedding → 向量库（保留）
图谱线：抽取实体 + 关系 → 构建知识图谱 → 沿图谱路径遍历查询
```

**典型场景**：用户问「哪些产品会受到半导体短缺影响？」
- Classic RAG 可能只找到含"半导体短缺"的文本片段
- Graph RAG 沿路径走：供应商 S → 芯片 Z → 电路板 Y → 产品 X，精准找出受影响范围

**适用场景**：影响分析、依赖分析、组织关系查询、供应链分析、因果链解释。

**代价**：建图成本高（需从文本中准确抽取实体和关系），维护成本高（业务变化需持续更新）。

> Graph RAG 解决的是「**信息之间如何连接**」的问题。

### 2.3 Agentic RAG — 推理：决定下一步

不再执行固定检索流程，而是让 **Agent 自主判断**：下一步该查什么、用什么工具、证据够不够、要不要重新规划。

```
问题 → Agent 拆解 → 选择工具/数据源 → 执行查询 → 评估结果
  → 证据不足？→ 换工具/换数据源 → 继续查询 → 综合判断 → 输出结论
```

**典型场景**：「为什么产品销量下降了？」Agent 可能需要依次查询销售数据库、价格历史、营销记录、客服工单、库存系统——证据不足时继续选择新工具。

**可调用的工具**：内部文档搜索、向量数据库、知识图谱查询、SQL 数据库、Web API、工单系统、日志系统……

**代价**：延迟高（多轮调用不可预测）、成本高（每次规划都消耗 token）、调试难。

> ⚠️ Agentic RAG 不是默认更高级。如果只是 FAQ，用 Agentic RAG 就像为拧一颗螺丝搬来一台挖掘机。

> Agentic RAG 解决的是「**面对复杂问题，下一步该怎么行动**」的问题。

### 2.4 三种架构对比与选型

| 维度 | Classic RAG | Graph RAG | Agentic RAG |
|------|------------|-----------|-------------|
| **核心动作** | 检索 | 连接 | 推理 |
| **适合问题** | 一跳问答、文档查询 | 依赖分析、影响分析 | 多步骤调查、跨系统分析 |
| **主要优势** | 快、简单、成本低 | 沿关系路径查找 | 灵活，动态选择工具 |
| **典型代价** | 不擅长关系链 | 建图维护成本高 | 延迟高、成本高、调试难 |

**选型心法：看问题形状，不看架构名字。**

```
第一问：答案能否一步命中？ → 可以 → Classic RAG（80% 的问题）
第二问：答案是否依赖关系链？ → 是 → 加入 Graph RAG
第三问：查询路径能否提前定义？ → 不能 → 考虑 Agentic RAG
```

**工程实践原则**：先用简单方案解决 80% 的问题，再为真正复杂的问题增加复杂度。

## 第三部分：RAG 优化管线（20 法）

原型一个下午就能跑通，但真正"能用"需要系统化优化。RAG 的本质不是让 AI 更聪明，而是**让正确的信息在对的时机出现在对的位置**。

```
数据入库 → 检索前（查询优化）→ 检索阶段 → 检索后（结果处理）→ 生成阶段
```

**每一个阶段的改进，都在给下一个阶段"减负"。**

### 3.1 数据入库（5 法）

| # | 方法 | 一句话 | 关键技巧 |
|---|------|--------|---------|
| 1 | **语义分块** | 别再按字符数切了 | 按句子、段落或 Markdown 标题切分 |
| 2 | **小块检索 + 大块投喂** | 图书索引法 | 小块做向量检索，命中后把整段喂给 LLM |
| 3 | **元数据过滤** | 先粗筛再细搜 | 存文档时顺手存时间、作者、类型，检索时先 `WHERE` |
| 4 | **摘要索引** | 让宏观问题先找到方向 | 长文档入库前让 LLM 生成摘要，单独向量化 |
| 5 | **文档反向提问** | 提前帮文档"写考题" | LLM 为每段文本生成 5 个假问题，和原文绑定入库 |

### 3.2 检索前优化（4 法）

用户的问题往往口语化、简略、甚至有歧义——直接拿去搜等于让口语碰书面语。

| # | 方法 | 一句话 | 关键技巧 |
|---|------|--------|---------|
| 7 | **查询重写** | 帮用户"翻译" | LLM 改写为标准化查询语言，**性价比极高** |
| 8 | **多路查询** | 一个问题五个角度 | 扩展为 3-5 个不同角度提问，分别检索合并去重 |
| 9 | **HyDE** | 先写假答案再搜真文档 | LLM 凭空写假答案→向量化→拿假答案搜真文档 |
| 10 | **查询路由** | 不是所有问题都走向量检索 | SQL 问题 → 数据库，闲聊 → 直接 LLM，语义查询 → 向量库 |

> HyDE 是最有意思的方法：「答案和答案」在向量空间的距离，往往比「问题和答案」更近。**用"假的"去找"真的"。**

### 3.3 检索阶段（4 法）

| # | 方法 | 一句话 | 盲区 |
|---|------|--------|------|
| 11 | **稠密检索** | 懂语义，不懂术语 | 专业术语、缩写、代码函数名 |
| 12 | **稀疏检索（BM25）** | 懂术语，不懂语义 | 同义词、改写表达 |
| 13 | **混合检索 + RRF** | 两条腿走路 | BM25 + 向量 + RRF 融合，盲区互补 |
| 14 | **微调嵌入模型** | 让领域术语不再"听不懂" | 用内部数据微调 Embedding 模型 |

**黄金组合**：稠密检索 + 稀疏检索 + RRF（倒数排名融合）。语义和关键词，一个都不能少。

### 3.4 检索后处理（3 法）

大多数人直接把 top_k 结果塞给 LLM，但检索回来的东西往往**有噪声、有重复、有废话**。

| # | 方法 | 一句话 | 为什么重要 |
|---|------|--------|-----------|
| 15 | **重排（Reranking）** | 性价比最高 | Cross-Encoder 逐字比对，对 top 20 重排就够了 |
| 16 | **上下文压缩** | 帮 LLM 划重点 | LLMLingua 等工具剔除废话，省 Token 还降幻觉 |
| 17 | **MMR** | 别全搜出同一句话 | 最大边际相关性，保证结果既相关又多样 |

> **重排是所有优化里性价比最高的**：只对初筛 top 20 做 Cross-Encoder 重排，投入小，效果大。

### 3.5 生成阶段（4 法）

| # | 方法 | 一句话 |
|---|------|--------|
| 18 | **Prompt 护栏** | 加一句"不知道就说不知道"，成本几乎为零 |
| 19 | **自我反思** | LLM 生成答案后再检查：说的内容是否真的来自上下文？ |
| 20 | **引用溯源** | 要求标注来源 `[1]` `[2]`，企业级 RAG 的标配 |

### 按场景选方案（组合 3-5 种）

| 场景 | 推荐组合 |
|------|---------|
| 通用知识问答 | 语义分块 → 混合检索 → 重排 |
| 企业私有文档 | 元数据过滤 → Graph RAG → 引用溯源 |
| 代码库问答 | 查询重写 → 稀疏检索 → 重排 |
| 长文档分析 | 摘要索引 → MMR → 上下文压缩 |

## 第四部分：知识图谱

### 4.1 核心概念：三元组、本体、实体消歧

知识图谱用 **实体（节点）+ 关系（边）** 来组织知识。

**三元组**是知识的基本形式：

```
乔布斯 → 创立 → 苹果公司
苹果公司 → 发布 → iPhone
iPhone → 属于 → 智能手机
```

**本体（Ontology）** 是图谱的架构设计图——规定有哪些实体类型、哪些关系类型、哪些关系合法。

**实体消歧**是第一项关键任务：当问题的关键不在单条数据，而在对象之间的关系时，知识图谱开始有价值。

```
数据库      → 存储精确记录
搜索引擎    → 找包含关键词的文本
向量数据库  → 找语义相似的内容
知识图谱    → 表达实体之间的关系
```

### 4.2 构建流程与实践

```
确定业务问题 → 设计 Schema/本体 → 接入数据源 → 抽取实体和关系
→ 实体链接与消歧 → 融合多源数据 → 质量校验 → 存储查询 → 接入应用 → 持续迭代
```

**第一版不追求大而全**：把核心实体和核心关系连准，比堆砌很多类型更重要。

**数据源分类**：
| 类型 | 示例 | 处理方式 |
|------|------|---------|
| 结构化 | 数据库、表格 | 规则直接映射 |
| 半结构化 | 网页、JSON | 解析和转换 |
| 非结构化 | 文档、邮件、工单 | NLP/大模型抽取，需验证 |

### 4.3 GraphRAG：图谱 + RAG

```
传统 RAG：文档切块 → 向量检索 → 召回文本 → 大模型生成答案
GraphRAG：识别实体和关系 → 构建图结构 → 沿图谱找上下文 → 大模型生成答案
```

知识图谱不替代大模型，而是**给大模型提供更稳固的事实骨架**。特别适合：跨多文档问题、多实体多跳查询、需要解释来源路径的答案。

### 4.4 代码知识图谱

当传统 IDE 全局搜索告诉你「这个函数被引用 12 次」，代码知识图谱能告诉你：从入口点到目标函数的**完整调用链**、改一个模块会**级联影响**哪些下游、某段代码**为什么**这样设计。

**与传统 RAG 的区别**：

| 维度 | 传统文档 RAG | 代码知识图谱 |
|------|-------------|-------------|
| **输入** | 文档文本 | 代码 AST + 文档 + 设计资料 |
| **切分** | 固定大小 chunk | 结构化节点（函数/类/模块） |
| **检索** | 语义相似度 | 关系遍历 + 语义搜索 |
| **回答** | 基于文本片段 | 基于调用链 + 依赖图 |

## 第五部分：LLM Wiki 范式

### 编译器类比

Karpathy 2026 年提出的 LLM Wiki 范式，核心类比极其精妙：

**AI 不是检索器，是编译器。**

```
raw/      = 源代码（不可变的原始资料）
LLM       = 编译器
wiki/     = 可执行输出（结构化知识页面）
lint      = 测试（发现矛盾、孤立页面）
queries   = 运行时（在知识库上提问）
```

### 三层架构

```
知识库/
├── raw/        # 不可变的原始资料（文章、论文、网页剪藏）
├── wiki/       # AI 生成和维护的 wiki 页面
│   ├── index.md    # 全局目录
│   └── log.md      # 操作日志
└── CLAUDE.md   # 模式文件：告诉 AI 如何操作这个 wiki
```

### 三个核心操作

| 操作 | 作用 | 类比 |
|------|------|------|
| **Ingest（摄取）** | 原始资料 → AI 提取信息整合到 wiki | 编译 |
| **Query（查询）** | 提问 → AI 读索引定位页面 → 综合回答 | 运行时 |
| **Lint（检查）** | 发现矛盾、孤立页面、断裂的交叉引用 | 测试 |

### LLM Wiki vs RAG：核心差异

| 维度 | RAG | LLM Wiki |
|------|-----|----------|
| **知识状态** | 无状态，每次从零发现 | 有状态，知识持续积累 |
| **维护成本** | 无（不需维护索引外的东西） | LLM 自动维护（交叉引用、矛盾检测） |
| **适用规模** | 百万级文档 | 个人/团队规模（~100 篇文章） |
| **基础设施** | 向量数据库 + Embedding | Git + Markdown |
| **知识复利** | 无积累 | 新知识建立在旧知识之上 |

> 📖 详细内容：[[../../Wiki/wiki/topics/obsidian-llm-wiki-practice|Obsidian LLM Wiki 实践]]

## 第六部分：向量数据库与嵌入模型

### 向量数据库选型（以 Milvus 为例）

| 场景 | 索引 | 原理 |
|------|------|------|
| 小数据，绝对精准 | **FLAT** | 暴力全比对 |
| 常规百万级向量 | **IVF_FLAT** | 聚类分桶后查桶内 |
| 海量内存紧张 | **IVF_SQ8** | 向量压缩省内存 |
| 高并发低延迟 | **HNSW** | 图结构跳跃查找 |
| 平衡速度与精度 | **SCANN** | IVF+SQ8+HNSW 组合 |
| 海量 SSD 低成本 | **DISKANN** | 向量落盘，内存只留导航 |

**相似度量三种**：L2（欧式距离，越小越相似）、IP（内积，越大越相似）、COSINE（余弦，看方向不看长度）。

### 嵌入模型选择

| 场景 | 推荐模型 | 特点 |
|------|---------|------|
| 纯文本语义检索 | OpenAI `text-embedding-3-small`、Jina `jina-embeddings-v3` | 通用 Dense |
| 关键词精确命中 | BM25、TF-IDF | 稀疏 Sparse |
| **工业级中文 RAG** | **BAAI/bge-m3** | 一次调用同时输出 dense + sparse + colbert |
| 多语言中文 | bge-m3 / Jina / OpenAI | 多语言 Dense |

> 参考 MTEB 排行榜：huggingface.co/spaces/mteb/leaderboard

## 推荐学习路线图

### 🟢 入门（1-2 天）

1. **先理解为什么**：阅读 [[../../Wiki/wiki/concepts/rag-architectures|RAG 三种架构]]，理解 Classic / Graph / Agentic 的本质差异
2. **搭一个原型**：用 ChromaDB + OpenAI Embedding 搭一个 Classic RAG 原型，感受"能跑"的状态
3. **读 LLM Wiki 原文档**：理解"编译器"类比（`Wiki/llm-wiki.md`）

### 🟡 进阶（3-5 天）

4. **系统化优化**：按 [[../../Wiki/wiki/topics/rag-optimization-techniques|RAG 优化 20 法]] 逐阶段优化你的原型
5. **理解知识图谱**：阅读 [[../../Wiki/wiki/concepts/knowledge-graph|知识图谱]]，手动构建一个小型领域图谱
6. **尝试 GraphRAG**：在原型中加入实体-关系抽取和知识图谱检索

### 🔴 深入（1-2 周）

7. **研究代码知识图谱**：安装 GitNexus 或 Graphify，在自己的项目中生成图谱
8. **实践 LLM Wiki**：按照 LLM Wiki 范式搭建自己的知识库（参考 [[../../Wiki/wiki/topics/obsidian-llm-wiki-practice|实践页面]]）
9. **探索 Agentic RAG**：当传统检索不够用时，引入 Agent 做多步骤自主推理

### 🚀 每个阶段的关键产出

| 阶段 | 关键产出 |
|------|---------|
| 入门 | 一个能跑的 Classic RAG 原型 |
| 进阶 | 优化到"能用"的 RAG + 一个小型知识图谱 |
| 深入 | 一份个人 LLM Wiki + 一个代码知识图谱项目 |

> **核心理念**：知识库技术的价值不在于技术本身，而在于它能否帮你**从"每次重新推导"升级到"持续积累知识"**。
