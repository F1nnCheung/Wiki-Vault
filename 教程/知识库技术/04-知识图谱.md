# 知识库技术 04 — 知识图谱与代码知识图谱

> 知识图谱不是"画一张漂亮的图"，而是把散落在文档、数据库和人脑里的**关系显式化**。当问题的关键不在单条数据，而在对象之间的连接时——知识图谱开始有价值。

---

## 为什么数据库不够用

### 四种技术定位

```
数据库      → 擅长存储精确记录
搜索引擎    → 擅长找包含关键词的文本
向量数据库  → 擅长找语义相似的内容
知识图谱    → 擅长表达实体之间的关系
```

### 一个案例看清差异

金融风控：单独看任何一张表，每个用户都正常。但把关系连起来会看到隐藏风险：

```
用户A → 使用 → 设备X ← 使用 ← 用户B → 使用 → 设备X ← 使用 ← 用户C
用户A → 绑定 → 银行卡Y ← 收款至 ← 用户B
用户A、B、C → 登录自 → 同一IP地址
```

**风险不在某一条数据里，而在关系构成的模式中。**

数据库可以通过 JOIN 查询关系，但当关系类型多、跳转层数深、数据来源杂时，表结构会变得复杂难维护。

> 知识图谱不是替代数据库，而是在数据库和应用之间增加一层"关系理解能力"。

---

## 核心概念三件套

### 三元组：知识的基本形式

```
乔布斯 → 创立 → 苹果公司
苹果公司 → 发布 → iPhone
iPhone → 属于 → 智能手机
北京 → 是首都 → 中国
```

一条三元组 = 一个事实。众多三元组连起来 = 知识图谱。

### 本体（Ontology）：图谱的架构图

本体规定了三件事：

| 规则 | 例子 |
|------|------|
| 有哪些实体类型？ | 人、公司、产品、地点 |
| 有哪些关系类型？ | 创立、发布、属于、位于 |
| 哪些关系是合法的？ | "人"可以有"出生地"，"公司"不能有 |

**没有本体约束，图谱容易变成一堆杂乱的连线。**

### 实体消歧：身份很关键

难点不在识别一个词，而在确认它到底指向哪个对象：

| 词 | 可能指向 |
|----|---------|
| "苹果" | 水果？苹果公司？内部项目代号？ |
| "张伟" | 销售部的张伟？客户联系人张伟？ |

**实体识别和实体消歧**是图谱的第一项关键任务。

---

## 构建流程（10 步）

```
确定业务问题 → 设计 Schema/本体 → 接入数据源
→ 抽取实体和关系 → 实体链接与消歧 → 融合多源数据
→ 质量校验 → 存储查询 → 接入应用 → 持续迭代
```

### 第一版原则

**不追求大而全**。把核心实体和核心关系连准，比堆砌很多类型更重要。

### 数据源分类

| 类型 | 示例 | 处理方式 |
|------|------|---------|
| 结构化 | 数据库、表格 | 规则直接映射（最可靠） |
| 半结构化 | 网页、JSON | 解析和转换 |
| 非结构化 | 文档、邮件、工单 | NLP/大模型抽取，需人工验证 |

## GraphRAG：图谱 + 大模型

### 与传统 RAG 的对比

```
传统 RAG：
文档切块 → 向量检索 → 召回文本 → 大模型生成答案
       ↑ 只能找"相似的文本"

GraphRAG：
识别实体和关系 → 构建图结构 → 沿图谱找上下文 → 大模型生成答案
       ↑ 能找"有关系的实体"，不依赖文本相似度
```

### 知识图谱的角色

知识图谱不替代大模型，而是**给大模型提供更稳固的事实骨架**。

特别适合：
- 跨多文档的问题
- 多实体、多跳关系查询
- 需要解释来源和路径的答案

### 应用场景

| 场景 | 核心价值 |
|------|---------|
| **搜索** | 从"字符串匹配"到"实体理解"（Google: "Things, not strings"） |
| **推荐** | 不只"买了A也买了B"，能理解商品类目、品牌关系 |
| **风控** | 发现隐藏在多跳关系里的团伙欺诈模式 |
| **医疗** | 疾病-症状-药物-基因天然适合图谱表达 |
| **企业 KM** | 项目、系统、接口、人员、文档、决策全连起来 |
| **GraphRAG** | 沿图谱找上下文→交给大模型生成答案 |

## 代码知识图谱

### 从"搜文件"到"查关系"

传统 IDE 全局搜索告诉你「这个函数被引用 12 次」。代码知识图谱能告诉你：
- 从入口点到目标函数的**完整调用链**
- 改一个模块会**级联影响**哪些下游
- 某段代码**为什么**这样设计（关联到设计文档）

### 为什么 AI 编程需要它

当前 AI 编程助手的困境：每次提问都要重新扫描文件。项目一大就开始吃力——**文件都在，理解没有累积。**

代码知识图谱提供**"项目记忆层"**：
1. 先读取项目，把代码结构组织成图谱
2. AI 回答问题时沿已有关系查找和推理
3. 改代码前先查图谱知道影响面

### 核心技术

| 技术 | 作用 | 工具 |
|------|------|------|
| AST 解析 | 提取函数、类、方法、导入关系 | Tree-sitter |
| 关系抽取 | 调用链、继承、类型依赖 | 跨文件分析 |
| 聚类分析 | 自动分组为功能社区 | Leiden 算法 |
| 混合搜索 | BM25 + 语义 + RRF | 多路融合 |

### 与传统 RAG 的区别

| 维度 | 传统文档 RAG | 代码知识图谱 |
|------|-------------|-------------|
| 输入 | 文档文本 | 代码 AST + 文档 |
| 切分 | 固定大小 chunk | 结构化节点（函数/类） |
| 检索 | 语义相似度 | 关系遍历 + 语义搜索 |
| 回答 | 基于文本片段 | 基于调用链 + 依赖图 |

---

## 核心洞见

> 知识图谱的真正价值不在于把数据画成一张网，而在于**把原本散落、隐含、难以复用的关系显式化**。

> 判断一个场景是否需要知识图谱：**当问题的关键不在单条数据，而在对象之间的关系；当答案不仅要给结果，还要能解释来源和路径**——知识图谱就开始有价值。
