# 知识库技术 06 — 向量数据库与嵌入模型选型

> 向量数据库是 RAG 的心脏，嵌入模型是 RAG 的眼睛。选错了心脏，查询慢如蜗牛；选错了眼睛，搜出来的东西牛头不对马嘴。

---

## 向量数据库核心概念

### 什么是向量数据库

```
传统数据库：存数字、字符串 → SELECT * WHERE name = '张三'
向量数据库：存向量（浮点数组） → SELECT * WHERE vector ≈ query_vector
```

**"≈"（近似）** 是关键——向量数据库做的是**近似最近邻搜索（ANN）**，不是精确匹配。

### 以 Milvus 为例的数据模型

```
Database（逻辑隔离）
  └── Collection（表）
       ├── Field: id (主键)
       ├── Field: title (文本)
       ├── Field: date (时间)
       └── Field: embedding (向量, 768维/1024维/1536维)
            └── Entity（行）= 一条记录
```

## 六大索引类型速查

索引决定了向量检索的速度和精度的平衡。

| 索引 | 适用场景 | 原理 | 精度 | 速度 | 内存 |
|------|---------|------|------|------|------|
| **FLAT** | < 10 万向量，要求绝对精准 | 暴力全比对 | ⭐⭐⭐ | ⭐ | ⭐⭐⭐ |
| **IVF_FLAT** | 百万级向量，常规场景 | 聚类分桶→只查最近几个桶 | ⭐⭐⭐ | ⭐⭐ | ⭐⭐ |
| **IVF_SQ8** | 海量向量，内存紧张 | IVF + 向量压缩省内存 | ⭐⭐ | ⭐⭐ | ⭐ |
| **HNSW** | 高并发低延迟 | 图结构跳跃查找 | ⭐⭐⭐ | ⭐⭐⭐ | ⭐⭐ |
| **SCANN** | 平衡型（推荐） | IVF + SQ8 + HNSW 组合 | ⭐⭐ | ⭐⭐ | ⭐ |
| **DISKANN** | 海量向量，SSD 低成本 | 向量落盘，内存只留导航 | ⭐⭐ | ⭐⭐ | ⭐ |

### 选型决策树

```
Q1: 向量数 < 10 万？
  → 是 → FLAT（反正全比对也很快）
  → 否 → Q2

Q2: 对延迟敏感（需要 < 10ms）？
  → 是 → HNSW
  → 否 → Q3

Q3: 内存紧张？
  → 是 → IVF_SQ8 或 DISKANN
  → 否 → IVF_FLAT 或 SCANN
```

## 相似度量三种

| 度量 | 公式含义 | 越相似越… | 适用场景 |
|------|---------|----------|---------|
| **L2（欧式距离）** | 两点直线距离 | 越小越相似 | 图像、通用 |
| **IP（内积）** | 向量投影长度 | 越大越相似 | 推荐系统 |
| **COSINE（余弦）** | 夹角余弦值 | 越大越相似 | 文本语义（最常用） |

**文本 RAG 首选 COSINE**——它只看方向，不管向量长度。两段文本表达同样的意思但篇幅不同时，余弦相似度不受影响。

## 嵌入模型选择

### 主流模型对比

| 模型 | 类型 | 维度 | 特点 | 适用场景 |
|------|------|------|------|---------|
| OpenAI `text-embedding-3-small` | Dense | 512/1536 | 通用最强，可调维度 | 英文文本，有预算 |
| **BAAI/bge-m3** | **Hybrid** | **1024** | **一次输出 dense+sparse+colbert** | **工业级中文 RAG（强烈推荐）** |
| Jina `jina-embeddings-v3` | Dense | 1024 | 多语言，8192 token 上下文 | 多语言文本 |
| Ollama `nomic-embed-text` | Dense | 768 | 本地运行，免费 | 隐私敏感、原型验证 |

### 为什么 bge-m3 是中文 RAG 的首选

一次 API 调用同时输出三种向量：

```
bge-m3("知识库技术") →
  ├── Dense vector (1024维)：语义理解
  ├── Sparse vector：关键词精确匹配
  └── ColBERT vector：token 级交互
```

**一个模型 = 稠密 + 稀疏 + 精细化**, 省去了维护两个模型的复杂度。

### 嵌入模型选型决策

```
Q1: 数据能出机器吗？
  → 不能 → Ollama 本地模型（nomic-embed-text / bge-m3）
  → 能 → Q2

Q2: 主要处理中文吗？
  → 是 → BAAI/bge-m3（混合检索一步到位）
  → 否 → Q3

Q3: 预算充足吗？
  → 是 → OpenAI text-embedding-3-small（质量最高）
  → 否 → bge-m3 或 Jina v3
```

## 实操：搭建最小 RAG 的向量层

### 用 ChromaDB（5 分钟原型）

```python
import chromadb
from chromadb.utils import embedding_functions

# 1. 初始化
client = chromadb.PersistentClient(path="./my_kb")
openai_ef = embedding_functions.OpenAIEmbeddingFunction(
    api_key="sk-xxx", model_name="text-embedding-3-small"
)

# 2. 创建 collection
collection = client.create_collection(
    name="my_docs", embedding_function=openai_ef
)

# 3. 入库
collection.add(
    documents=["Obsidian 是一个本地 Markdown 笔记工具……"],
    metadatas=[{"source": "obsidian-intro.md", "date": "2026-05-28"}],
    ids=["doc_1"]
)

# 4. 查询
results = collection.query(
    query_texts=["如何管理个人知识？"],
    n_results=5
)
```

### 用 Milvus（生产环境）

```python
from pymilvus import MilvusClient

client = MilvusClient("milvus_demo.db")

# 创建 collection（指定索引类型）
client.create_collection(
    collection_name="demo",
    dimension=1024,
    metric_type="COSINE",
    index_type="HNSW",     # 高并发低延迟
    params={"M": 16, "efConstruction": 200}
)

# 插入数据
client.insert(collection_name="demo", data=[{
    "id": 1,
    "vector": [...],  # 1024 维向量
    "text": "知识库技术……",
    "source": "tutorial-ch6.md"
}])

# 搜索
results = client.search(
    collection_name="demo",
    data=[query_vector],
    limit=5,
    output_fields=["text", "source"]
)
```

## 本章小结

1. 文本 RAG 首选 **COSINE** 相似度 + **bge-m3** 嵌入模型（一次输出三种向量）
2. 原型用 **ChromaDB**（零配置），生产用 **Milvus**（高性能）
3. **HNSW** 索引是大多数场景的最佳平衡点
4. 数据不能出机器 → Ollama + nomic-embed-text 本地方案
5. 参考 MTEB 排行榜了解最新模型表现：huggingface.co/spaces/mteb/leaderboard
