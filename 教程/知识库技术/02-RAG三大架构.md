# 知识库技术 02 — RAG 三大架构深度对比

> RAG（Retrieval-Augmented Generation）的核心思路只有一句话：**先把相关资料找出来，再让大模型基于这些资料回答**。但"怎么找"决定了系统的能力上限。本章拆解三种检索架构的本质差异。

---

## 先理解 RAG 的基本流程

任何 RAG 系统都包含两个阶段：

### 离线阶段（入库）

```
原始文档 → 清洗 → 切块（Chunk）→ Embedding（向量化）→ 存入向量数据库
```

### 在线阶段（查询）

```
用户问题 → Embedding → 向量检索 Top K → 拼入 Prompt → LLM 生成答案
```

### 为什么需要 RAG

LLM 有两个根本问题：
1. **知识截止**：训练数据停在某个时间点
2. **幻觉**：不知道的事情会编造

RAG 的解决方案：**不给 LLM 靠记忆回答，而是给它"开卷考试"的参考资料。**

---

## 架构一：Classic RAG — 找资料

### 一句话

把用户问题和文档都变成向量，用向量相似度找到最相关的内容片段。

### 完整流程

```
┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐
│  原始文档  │ → │  Chunk   │ → │ Embedding │ → │  向量库   │
└──────────┘    └──────────┘    └──────────┘    └──────────┘
                                                      │
                                                （离线完成）
╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌
                                                （在线查询）
                                                      │
┌──────────┐    ┌──────────┐    ┌──────────┐          │
│ 用户问题  │ → │ Embedding │ → │ Top K 检索│ ← ─ ─ ─ ┘
└──────────┘    └──────────┘    └──────────┘
                                      │
                                      ▼
                               ┌──────────┐
                               │ 拼入 Prompt│
                               └──────────┘
                                      │
                                      ▼
                               ┌──────────┐
                               │ LLM 生成  │
                               └──────────┘
```

### 适用场景

| 场景 | 为什么 | 例子 |
|------|--------|------|
| FAQ 问答 | 答案在固定文档中 | "退货政策是什么？" |
| 产品手册 | 信息结构化 | "这个功能怎么配置？" |
| 客服知识库 | 一问一答模式 | "怎么重置密码？" |
| 政策查询 | 权威文档 | "年假天数怎么算？" |

### 优势

- **简单**：一个下午能搭出原型
- **快**：毫秒级检索，无额外 LLM 调用
- **便宜**：只消耗 Embedding + 一次 LLM 调用
- **可解释**：能明确告诉用户"答案来自哪个文档"

### 本质局限：不懂关系

**经典翻车场景**：用户问「A 的经理的经理是谁？」

- 文档片段 A：「张三是李四的经理」
- 文档片段 B：「王五是张三的经理」
- Classic RAG 可能只召回片段 A，LLM 只能回答「李四」——但正确答案是「王五」

答案不在一个段落里，而在**关系链**中。

> Classic RAG 解决：「**资料在哪里**」

---

## 架构二：Graph RAG — 找关系

### 一句话

在文本检索之外加一层知识图谱，把"谁和谁有关系"显式建模出来。

### 双线架构

```
文本线（保留）：切块 → Embedding → 向量库
图谱线（新增）：抽取实体+关系 → 构建知识图谱 → 沿图谱路径查询
```

### 实体与关系

```
实体 = 节点：人、产品、部门、系统、供应商……
关系 = 边：  汇报给、属于、依赖、采购自……
```

### 典型场景：供应链影响分析

用户问「哪些产品会受到半导体短缺影响？」

| 方法 | 过程 | 结果 |
|------|------|------|
| Classic RAG | 找到包含"半导体短缺"的文本 | 可能提到某几个产品，但不完整 |
| Graph RAG | 供应商 S → 芯片 Z → 电路板 Y → 产品 X（沿图谱路径遍历） | 精准列出所有受影响产品 |

### 适用场景

- **影响分析**：一个变更影响哪些下游？
- **依赖分析**：某个组件被哪些对象依赖？
- **组织关系**：跨部门审批链查询
- **供应链追溯**：从原材料到成品

### 代价

| 代价 | 说明 |
|------|------|
| **建图成本高** | 从文本中准确抽取实体和关系需要模型能力，还可能需要人工校验 |
| **维护成本高** | 业务变化 → 图谱需持续更新，否则信息过时 |
| **边界限制** | 只能遍历已建模的节点和边，未建模的领域无路可走 |

### 不是替代，是增强

Graph RAG 不替代 Classic RAG——**它是在 Classic RAG 解决不了"关系链"问题时才上场的。**

> Graph RAG 解决：「**信息之间如何连接**」

## 架构三：Agentic RAG — 自主推理

### 一句话

不再执行固定的检索流程，让 AI Agent 自己判断：下一步该查什么、用什么工具、证据够不够。

### 动态决策循环

```
     ┌──────────────────────────────────┐
     │           用户问题                 │
     └──────────────┬───────────────────┘
                    ▼
     ┌──────────────────────────────────┐
     │   Agent 拆解：需要查什么？          │ ← ─ ─ ─ ─ ─ ─ ┐
     └──────────────┬───────────────────┘                 │
                    ▼                                     │
     ┌──────────────────────────────────┐                 │
     │   选择工具：向量库？SQL？知识图谱？   │                 │
     └──────────────┬───────────────────┘                 │
                    ▼                                     │
     ┌──────────────────────────────────┐                 │
     │   执行查询，拿到结果                │                 │
     └──────────────┬───────────────────┘                 │
                    ▼                                     │
     ┌──────────────────────────────────┐                 │
     │   证据够吗？                       │ ── 不够 ── → ─ ┘
     └──────────────┬───────────────────┘
                    │ 够了
                    ▼
     ┌──────────────────────────────────┐
     │   综合判断，输出结论               │
     └──────────────────────────────────┘
```

### 典型场景：销量下降归因

用户问「为什么产品销量下降了？」

Agent 的自主推理过程：

```
Step 1: 查销售数据库 → 确认 3 月销量环比下降 15%
Step 2: 查价格历史 → 价格没调
Step 3: 查营销投放记录 → 3 月投放减少了 40%
Step 4: 查客服工单 → 新版本发布后投诉增加了 3 倍
Step 5: 查库存系统 → 部分 SKU 缺货
Step 6: 综合判断 → 主要原因：营销减少 + 新版本质量问题 + 部分缺货
```

没有固定资料入口——Agent 需要自主选择先查什么、再查什么。

### Agent 的工具箱

| 工具类型 | 具体工具 | 用途 |
|---------|---------|------|
| 文档检索 | 向量数据库 | 找相关文本 |
| 关系查询 | 知识图谱 | 找实体关系链 |
| 结构化查询 | SQL 数据库 | 精确数据（销量、库存） |
| 实时数据 | Web API | 天气、汇率、新闻 |
| 业务系统 | 工单系统、CRM | 客户反馈、服务记录 |
| 日志分析 | ELK、Grafana | 系统故障排查 |

### 代价

| 代价 | 严重程度 |
|------|---------|
| **延迟高** | 🔴 多轮 LLM 调用，每次几百毫秒到几秒 |
| **成本高** | 🔴 每次规划、判断、重规划都消耗 token |
| **调试难** | 🟡 不仅要看检索结果，还要看 Agent 为什么做这个决策 |
| **不确定性** | 🟡 同样的输入可能走不同的查询路径 |

### 关键心法

> ⚠️ **Agentic RAG 不是默认更高级**。如果只是 FAQ，用 Agentic RAG 就像为拧一颗螺丝搬来一台挖掘机。

> Agentic RAG 解决：「**面对复杂问题，下一步该怎么行动**」

## 三种架构对比总表

| 维度 | Classic RAG | Graph RAG | Agentic RAG |
|------|------------|-----------|-------------|
| **一句话** | 找资料 | 找关系 | 自主推理 |
| **核心机制** | 向量相似度 | 知识图谱遍历 | Agent 动态规划 |
| **查询路径** | 固定 | 沿图谱 | 动态决策 |
| **适合问题** | 一跳问答、FAQ | 依赖分析、影响分析 | 多步调查、跨系统分析 |
| **延迟** | 毫秒级 | 毫秒-秒级 | 秒-分钟级 |
| **成本** | 低 | 中 | 高 |
| **建维成本** | 低 | 高 | 中 |
| **可解释性** | 高 | 高 | 中 |

## 三问选型法

不需要背架构名字，面对实际问题时问自己三句话：

```
第一问：答案能否一步命中（就在某个段落里）？
  → 能 → Classic RAG ← 80% 的问题都在这
  → 不能 → 继续第二问

第二问：答案的关键是不是"谁和谁的关系"？
  → 是 → 加 Graph RAG
  → 不是 → 继续第三问

第三问：查询路径能否提前确定？
  → 能 → Classic RAG + 分步查询就够了
  → 不能（需要看情况决定查什么）→ Agentic RAG
```

### 工程实践原则

```
1. 先上 Classic RAG → 覆盖 80% 的高频简单问题
2. 遇到关系链问题 → 引入 Graph RAG
3. 遇到开放式多步骤跨系统问题 → 交给 Agentic RAG
```

**不要在第一步就追求最复杂的方案。**

---

## 本章小结

1. **Classic RAG** = 向量相似度找资料，适合"答案在段落里"的问题
2. **Graph RAG** = 知识图谱找关系，适合"答案在关系链中"的问题
3. **Agentic RAG** = AI 自主决定查什么，适合"一开始不知道查什么"的问题
4. **选型心法**：看问题形状不看架构名字，先简单后复杂
