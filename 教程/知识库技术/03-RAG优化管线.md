# 知识库技术 03 — RAG 优化管线 20 法

> 一个下午能搭出能跑的 RAG 原型。但搜出来的东西牛头不对马嘴、明明库里有的内容就是找不到、问得稍抽象一点就开始编——这就是"能跑"和"能用"的差距。RAG 的本质不是让 AI 更聪明，而是**让正确的信息在对的时机出现在对的位置**。

---

## 五阶段管线

```
数据入库 → 检索前（查什么）→ 检索中（怎么查）→ 检索后（怎么筛）→ 生成（怎么答）
```

**核心原则**：每一个阶段的改进，都在给下一个阶段"减负"。入口数据差了，后面再强也白搭。

---

## 阶段一：数据入库（1-6）

### 1. 语义分块 — 别再按字符数切了

按 500 字固定长度切块，就像把一本书每 10 页拆下来订成一册——刚好把完整论证拦腰截断。

**正确做法**：按句子、段落或 Markdown 标题来切，保证每块是一个完整的逻辑单元。

```
❌ 固定大小：第 1 块 500 字，第 2 块 500 字……
✅ 语义分块：按 ## 标题切，每个标题下内容为一块
```

### 2. 小块检索 + 大块投喂 — 图书索引法

文档切成**很小片段**做向量检索（小块精度高），命中后把所在"整段大哥"调出来喂给 LLM。

> 就像图书索引：你用关键词查到页码，但真正读的是那一整页内容。

### 3. 元数据过滤 — 先粗筛再细搜

存文档时顺手存上时间、作者、类型。检索时先 `WHERE` 一下再做语义匹配。

```
用户："2024 年以后的 Python 异步文章"

❌ 纯向量检索：在百万级向量中匹配
✅ 先 WHERE date >= 2024 AND tag = "python" → 缩小到千级 → 再向量检索
```

**一个简单的条件过滤就能把搜索范围从百万级缩到千级。**

### 4. 摘要索引 — 让宏观问题先找到方向

长文档被切碎后，用户问"这本书讲了什么"时任何一个片段都答不上来。

**做法**：入库前让 LLM 给每个长文档生成全局摘要，单独向量化。宏观问题命中摘要 → 顺藤摸瓜找到细节。

### 5. Graph RAG — 不只搜"内容"，还搜"关系"

把文档里的实体和关系抽取出来存进知识图谱。检索时**同时查向量库和图数据库**。

详见 [[04-知识图谱|第 4 章：知识图谱]]。

### 6. 文档反向提问 — 提前帮文档"写考题"

入库前让 LLM 看着每段文本生成 5 个"如果有人要查这段内容，可能会怎么问"。把这些问题和原文绑在一起存入向量库。

> **相当于提前把所有可能的问法都做好了索引。** 这是入库阶段性价比最高的优化。

---

## 阶段二：检索前优化（7-10）

用户的提问往往口语化、简略、甚至有歧义——**直接拿去搜等于让口语问题去碰书面语文档**。

### 7. 查询重写 — 帮用户"翻译"

调用 LLM 将用户自然语言改写为更适合检索的标准问法。

```
用户：今年一二月大概赚了多少钱，各项收入和纯利润分别是多少呀？
改写：今年1月至2月的净利润和各项收入明细是多少？
```

**性价比极高**：多了一步 LLM 调用，但省去了后续大量试错。

### 8. 多路查询 — 一个问题五个角度

把用户问题扩展成 3-5 个不同角度的提问，分别检索，最后合并去重。

> **用广度换精度。**

### 9. HyDE — 先写假答案再搜真文档

这是最精妙的方法：

```
用户问题 → LLM 凭空写一个"假答案" → 把假答案向量化 → 用假答案去搜真文档
```

**为什么有效**？"答案和答案"在向量空间的距离，往往比"问题和答案"更近。**用"假的"去找"真的"。**

### 10. 查询路由 — 不是所有问题都走向量检索

| 问题类型 | 路由到 |
|---------|--------|
| "今年第二季度销售额？" | SQL 数据库 |
| "这段代码为什么报错？" | 向量库 |
| "你好，今天天气怎么样？" | 直接给 LLM |

**先分类，再处理。** 让正确的问题走正确的路。

---

## 阶段三：检索阶段（11-14）

### 11. 稠密检索（Dense）— 懂语义，不懂术语

基于 Embedding 的语义搜索。搜"轿车"能找到"小汽车""私家车"。

**盲区**：专业术语、缩写、代码函数名。搜"OOM Killer"可能找不到确切匹配。

### 12. 稀疏检索（Sparse/BM25）— 懂术语，不懂语义

基于关键词匹配。搜"OOM Killer"绝对不会漏掉相关文档。

**盲区**：搜不到表达方式不同导致的关键词不匹配。

### 13. 混合检索 + RRF — 两条腿走路

把稠密和稀疏结果用 **RRF（倒数排名融合）** 算法合并：

```
RRF_score(d) = Σ 1/(k + rank_i(d))
```

**黄金组合**：语义和关键词，一个都不能少。两种方法的盲区刚好互补。

### 14. 微调嵌入模型 — 让领域术语不再"听不懂"

通用 Embedding 模型对私有领域理解不好？用自己的数据微调一次。

> 比如用内部代码文档训 BGE，它就能理解你项目中的特有函数名和术语。

---

## 阶段四：检索后处理（15-17）

大多数人直接把 top_k 结果塞给 LLM。但检索回来的东西往往**有噪声、有重复、有废话**。

### 15. 重排（Reranking）— 性价比最高的优化

| 方法 | 原理 | 速度 | 精度 |
|------|------|------|------|
| 双塔模型（向量检索） | 查询和文档分别编码 | 快 | 中 |
| Cross-Encoder（重排） | 查询和文档拼在一起逐字比对 | 慢 | 高 |

**最佳实践**：初筛 top 20 用快速向量检索 → 对这 20 个用 Cross-Encoder 重排。

> **重排是所有优化里性价比最高的**——投入小（只对 20 个做），效果大（排序质量质的飞跃）。

### 16. 上下文压缩 — 帮 LLM 划重点

用 LLMLingua 等工具把检索段落里的废话剔除，留核心信息。**既省 Token，又降 LLM "分心"概率。**

### 17. MMR — 别全搜出同一句话

检索出 5 段文本，4 段说的是同一回事——上下文窗口被白白浪费。

MMR（最大边际相关性）算法保证结果**既相关又多样**：

```
MMR = argmax[ λ·similarity(D_i, Q) - (1-λ)·max similarity(D_i, D_j) ]
                                    ↑ 惩罚与已选结果太相似的文档
```

---

## 阶段五：生成阶段（18-20）

### 18. Prompt 护栏 — 成本几乎为零

在 System Prompt 里加一句：

> "请严格基于提供的 Context 回答。如果 Context 中没有相关信息，请直接说'根据现有资料无法回答'，不要编造。"

**成本几乎为零，是所有优化里性价比最高的一项。**

### 19. 自我反思 — 答完之后回头看一眼

LLM 生成答案后再检查：说的内容是否确实来自给定上下文？如果没有，打回重写。

在关键业务场景下，宁可多花几秒也别让幻觉毁掉用户信任。

### 20. 引用溯源 — 企业级 RAG 的标配

让模型在回答里标注来源 `[1]` `[2]`。当模型知道自己的每句话都要"对得上账"时，它会回答得更谨慎。

```
示例输出：
2024 年营收增长 15% [1]，主要驱动因素是……
[1] 2024 年度财务报告，第 3 页
```

---

## 按场景组合（3-5 种即可）

| 场景 | 推荐组合 |
|------|---------|
| 通用知识问答 | 语义分块 → 混合检索 → 重排 |
| 企业私有文档 | 元数据过滤 → Graph RAG → 引用溯源 |
| 代码库问答 | 查询重写 → 稀疏检索 → 重排 |
| 长文档分析 | 摘要索引 → MMR → 上下文压缩 |
| 客服 FAQ | 反向提问 → 稠密检索 → Prompt 护栏 |

---

## 核心心法

1. **入库打好地基**：语义分块 > 字符分块；小块检索 + 大块投喂是黄金组合
2. **检索前帮用户翻译**：查询重写 + HyDE 假答案搭桥
3. **检索时两条腿走路**：稠密 + 稀疏混合检索
4. **检索后别偷懒**：重排是性价比最高的优化
5. **生成时设好护栏**：Prompt 加一句"不知道就说不知道"

> 不是所有场景都需要 20 种全上。理解你的语义鸿沟最大在哪一环，选 3-5 种对症的组合，就足够让"能跑"的 RAG 变成"能用"的 RAG。
