# 2.5.3 Codex 适用场景

> Codex 不是万能工具——它在特定场景下效率极高，在不适合的场景下可能浪费大量 Token 还拿不到好结果。本章帮你判断「什么时候该用 Codex」。

---

## 适合 Codex 的场景

### ✅ 1. 从零搭建新项目

**为什么适合**：新项目没有历史包袱，Codex 可以从头规划架构。

```
示例："创建一个 Next.js 14 + Prisma + NextAuth 的博客项目"
```

Codex 能自动完成：项目初始化、数据库 schema、认证配置、基础页面、部署配置。

### ✅ 2. 批量代码迁移/重构

**为什么适合**：重复性高、规则明确，Codex 可以自主批量处理。

```
示例："把 src/ 下所有 JavaScript 文件迁移到 TypeScript"
示例："把 class 组件改成函数组件 + Hooks"
示例："把所有 CSS 文件迁移到 Tailwind CSS"
```

### ✅ 3. 自动化脚本和工具

**为什么适合**：需求明确，边界清晰，不需要理解复杂业务逻辑。

```
示例："写一个脚本，每天自动备份数据库到 S3"
示例："写一个 CLI 工具，批量压缩 images/ 下的图片"
示例："写一个 GitHub Action，PR 合并时自动部署到 Vercel"
```

### ✅ 4. 生成文档和测试

**为什么适合**：规则明确，Codex 可以自动扫描代码并生成。

```
示例："给 src/api/ 下所有接口生成 JSDoc 文档"
示例："给 src/utils/ 下所有函数写单元测试，覆盖率 >80%"
示例："生成 API 参考文档（OpenAPI 格式）"
```

---

## 不适合 Codex 的场景

### ❌ 1. 与现有复杂项目深度集成

**为什么不合适**：Codex 缺乏对历史业务逻辑的深入理解，容易引入 Bug。

```
反例："给这个 3 年的电商项目加上优惠券系统"
  → Codex 不理解现有的订单流程、支付逻辑、库存管理
  → 大概率引入隐藏 Bug
```

**替代方案**：用 Claude Code + CLAUDE.md 在本地深入理解项目后逐步实现。

### ❌ 2. 需要精细业务逻辑的场景

**为什么不合适**：业务规则往往没有文档，靠人工判断。

```
反例："根据业务需求调整定价策略"
  → 「定价策略」的规则在人脑里，不在代码里
  → Codex 无法理解
```

**替代方案**：人工分析 + Cursor/Copilot 辅助编码。

### ❌ 3. 强依赖内部服务的系统

**为什么不合适**：Codex 无法访问内网服务、私有 API。

```
反例："对接公司内部用户中心的认证接口"
  → Codex 看不到接口文档，也无法测试
```

**替代方案**：手动开发 + ChatGPT/Cursor 辅助。

### ❌ 4. 安全性要求极高的代码

**为什么不合适**：安全相关代码必须经过严格人工审查。

```
反例："实现支付回调验证逻辑"
  → 涉及资金安全，任何 Bug 都可能造成严重后果
```

**替代方案**：人工开发 + AI 辅助写测试。

## 场景决策矩阵

```
你的任务是什么类型？
├── 从零搭新项目 → ✅ Codex
├── 批量迁移/重构 → ✅ Codex
├── 自动脚本/工具 → ✅ Codex
├── 生成文档/测试 → ✅ Codex
├── 修改已有复杂项目 → ❌ 用 Claude Code
├── 精细业务逻辑 → ❌ 人工 + Cursor
├── 依赖内部服务 → ❌ 人工开发
└── 安全关键代码 → ❌ 人工 + 严格审查
```

---

## Codex 在工具组合中的位置

Codex 不是日常主力工具，而是特定场景的「特种兵」：

```
日常编码 80%：Trae（国内）/ Cursor（海外）
  +
架构攻坚 15%：Claude Code
  +
自动化脚本 4%：Codex ← 这就是 Codex 的位置
  +
个人助手 1%：OpenClaw / Hermes
```

---

## 使用 Codex 的最佳实践

1. **先小后大**：先用简单任务测试，确认 Codex 理解你的项目
2. **写 Spec**：花 10 分钟写规格文档，节省数小时返工
3. **分步执行**：大任务拆成多个小任务，逐步验证
4. **独立分支**：让 Codex 在独立分支上工作，Review 后再合并
5. **检查依赖**：Codex 可能安装过多或不安全的依赖，务必检查 package.json 变更

---

## 本章小结

1. Codex 最适合：新项目搭建、批量迁移、自动化脚本、文档测试生成
2. Codex 不适合：复杂项目修改、精细业务逻辑、依赖内部服务、安全关键代码
3. Codex 定位：4% 的自动化场景，不是主力工具
4. 关键习惯：写 Spec、分步执行、独立分支、Review 每个 PR
