# 2.2.3 Claude Code 基本使用

> 掌握核心命令和日常工作流，是 Claude Code 从「装了」到「会用」的关键一步。本章覆盖新手必备命令、每日工作流和权限管理。

---

## 概述

Claude Code 有 50+ 个命令，但多数人日常只需要 10 个。关键是理解每个命令的「何时用」和「为什么用」。

---

## 新手必知 6 个命令

如果你是第一次使用 Claude Code，先记住这 6 个：

| 命令 | 作用 | 什么时候用 |
|---|---|---|
| `/help` | 查看帮助 | 不知道怎么办时——这是你的安全网 |
| `/clear` | 清空上下文 | 话题完全切换时 |
| `/compact` | 压缩上下文 | 上下文占用 70%+ 时（保留主线逻辑） |
| `/model` | 切换模型 | 想换模型时 |
| `/effort` | 调整推理深度 | 简单任务降配省钱、复杂任务升配保质量 |
| `/permissions` | 管理权限规则 | 权限弹窗太多时 |

---

## 每日 Top 10 命令

按使用频率从高到低排列：

### 1. `/init` — 生成项目 CLAUDE.md

**何时用**：新项目启动时，第一件事。

```bash
/init
```

自动分析项目结构，生成 CLAUDE.md 初始版本。之后用 `/memory` 手动补充。

### 2. `/compact` — 压缩上下文

**何时用**：上下文占用 70-80% 时。

> ⚠️ **这是最被低估的命令**。多数人条件反射式 `/clear`，但 `/compact` 保留主线逻辑，是更优选择。

```
/clear  = 扔掉所有记忆，从头开始
/compact = 保留关键信息，压缩冗余
```

### 3. `/clear` — 清空会话

**何时用**：话题彻底切换，和之前的工作完全无关时。

### 4. `/model` — 查看和切换模型

**何时用**：需要确认当前使用的模型，或切换到更便宜/更强的模型。

```bash
/model              # 查看可用模型
/model Opus 4.6    # 切换到最强模型
/model Haiku       # 切换到便宜模型做简单任务
```

### 5. `/cost` — 查看 Token 消耗

**何时用**：担心费用时，或做了大量操作后想了解消耗情况。

### 6. `/context` — 查看上下文占用

**何时用**：感觉 AI 开始「健忘」或响应变慢时。

```
上下文占用 < 50%：正常
上下文占用 50-70%：注意，即将需要 compact
上下文占用 70-80%：立即 compact
上下文占用 > 90%：AI 可能已经开始遗忘早期对话
```

### 7. `/diff` — 查看代码改动

**何时用**：提交代码前必看。确保 AI 没有意外修改不该改的东西。

### 8. `/memory` — 编辑 CLAUDE.md

**何时用**：发现 AI 反复犯同一个错误时，把规则写入 CLAUDE.md。

### 9. `/resume` — 恢复历史会话

**何时用**：会话中断后想继续之前的工作。

```bash
# CLI 方式
claude -c    # 续接上次会话
```

### 10. `/rewind` — 回退操作

**何时用**：AI 做错了，想撤销最近的操作。

快捷键：Esc Esc

## 日常工作流

一个完整的日常开发流程：

```
第一步：规划模式
  ↓ 输入 /plan → 描述需求，只讨论不修改
  ↓ 确认方案可行
第二步：执行
  ↓ 切换到默认/自动模式 → AI 开始写代码
  ↓ 过程中用 /context 监控上下文占用
第三步：压缩
  ↓ 上下文占用 70%+ → /compact
第四步：检查
  ↓ /diff 查看所有改动
  ↓ 确认无误
第五步：提交
  ↓ git add、commit
```

### 实操示例

```
你：/plan
你：我要实现用户注册功能，包含邮箱验证和密码强度检测

AI：[分析现有代码库...]
AI：我建议分四步：
    1. 创建 User model (Prisma)
    2. 实现 /api/auth/register 接口
    3. 添加邮箱验证逻辑
    4. 写单元测试
    可行吗？

你：可以，但第三步用 Resend 发验证邮件

AI：明白。那我开始实现

[AI 创建文件、写代码...]

你：/context    ← 检查上下文
你：/diff       ← 检查改动
你：/cost       ← 查看费用
```

## 权限管理三原则

Claude Code 对所有操作都需要权限确认。分三个级别：

| 级别 | 含义 | 典型操作 | 策略 |
|---|---|---|---|
| **allow** | 自动允许 | lint、test、git status | 低风险高频操作 |
| **ask** | 每次询问 | 修改文件、安装依赖 | 中风险操作 |
| **deny** | 禁止 | .env 文件、git push --force、rm -rf | 高风险操作 |

### permissions 配置示例

```json
{
  "permissions": {
    "allow": [
      "Bash(npm test:*)",
      "Bash(npm run lint:*)",
      "Bash(git status:*)",
      "Bash(git diff:*)"
    ],
    "deny": [
      "Bash(git push --force:*)",
      "Bash(rm -rf:*)",
      "Read(.env:*)",
      "Edit(.env:*)"
    ]
  }
}
```

### 权限管理原则

1. **先严后松**：刚开始全部用 ask，确认安全后逐步 allow
2. **生产环境更严**：deny 所有 git push --force、数据库删除等操作
3. **`.env` 永远 deny**——绝不让 AI 读取或修改密钥文件

## 上下文与会话管理

### 何时 compact vs clear？

```
/compact 优先于 /clear

✅ /compact 适用：
- 上下文占用高但仍需继续当前话题
- AI 输出质量下降但任务还没完成
- 想保留主线逻辑但释放空间

✅ /clear 适用：
- 话题完全切换
- 上下文混乱，AI 出现了幻觉
- 开始一个全新项目
```

### 上下文监控习惯

养成每次大操作后 `/context` 的习惯：
- 读了一个大文件后 → 检查
- AI 完成了一个复杂任务后 → 检查
- 感觉 AI 开始「失忆」→ 立即检查

## CLI 启动标志

除了在对话中使用命令，还可以在启动时指定行为：

```bash
claude --print "分析这个项目的架构"  # 单次执行，输出结果后退出
claude -c                             # 续接上次会话
claude --dangerously-skip-permissions # 完全自主模式（慎用）
```

## 本章小结

1. 新手先记住 6 个命令：help、clear、compact、model、effort、permissions
2. 每日工作流：规划 → 执行 → 压缩 → 检查 → 提交
3. **`/compact` > `/clear`**——保留主线优于重新开始
4. 权限管理：先严后松，`.env` 永远 deny
5. 上下文 70%+ 必须 compact
