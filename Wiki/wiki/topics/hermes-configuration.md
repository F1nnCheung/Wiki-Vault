# Hermes 配置指南：SOUL.md & AGENTS.md

很多人装完 Hermes 第一反应是去找 Skill。但常见的「反复问同一个背景」「文件乱放」「改完代码不测试」等问题，往往不是 Skill 不够，而是没给它写规则。

Hermes 里最值得先处理的，不是装多少插件，而是两个文件：**SOUL.md** 和 **AGENTS.md**。

> 一个管「它怎么和你长期协作」，一个管「它在这个项目里该怎么干活」。

## 一、两个文件的分工

| 文件 | 主要作用 | 放在哪里 |
|------|---------|---------|
| `SOUL.md` | 全局身份、语气、协作方式 | `~/.hermes/SOUL.md` 或 `$HERMES_HOME/SOUL.md` |
| `AGENTS.md` | 项目规则、命令、目录、注意事项 | 项目根目录 |

**一句话**：
- `SOUL.md` — 给 Hermes 定「性格」
- `AGENTS.md` — 给项目定「规矩」

前者适合长期跟着你走，后者只服务当前项目。

## 二、SOUL.md 详解

`SOUL.md` 是 Hermes 的主要身份文件。它不是项目说明书，而是你给 Hermes 写的一份长期协作说明：

- 你希望它说话直接还是温和
- 你希望它先给结论还是先铺背景
- 遇到不确定信息时要不要主动核实
- 高风险操作前要不要先提醒
- 输出内容要克制还是更有表达感

> ⚠️ 常见错误：把项目路径、端口、启动命令、接口说明塞进 `SOUL.md`。这些不该放这里。`SOUL.md` 只写全局习惯。

### 可复制模板

```markdown
# Identity

你是我的长期 AI 协作助手。

你的主要任务是帮助我完成：
- 资料整理
- 代码辅助
- 工具配置
- 自动化任务
- 项目排障

你的目标不是显得聪明，而是把问题处理清楚。

# Communication Style

默认使用中文。
先说结论，再解释原因。
能用短句就不用长句。
不确定时直接说"不确定"，不要编造答案。
如果信息可能过期，优先提醒我核实官方文档、GitHub 或 README。

# Working Principles

开始处理任务前，先理解当前目录、现有文件和已有规则。
不要主动修改无关文件。
不要在没有确认的情况下删除、覆盖、迁移、发布或执行高风险操作。
涉及安装依赖、改配置、迁移数据、连接外部服务时，先说明影响。

# Output Preferences

输出要直白、克制、实用。
少讲空泛道理，多给可执行步骤。
写教程时，按"适合谁、解决什么问题、怎么做、有什么坑"来组织。

# Avoid

不要夸大工具能力。
不要承诺收益、涨薪、转型成功或副业结果。
不要把猜测写成事实。
不要用未经核实的数据做确定判断。
```

## 三、AGENTS.md 详解

`AGENTS.md` 管的是：这个项目里，什么能做，什么不能做。

关键信息：
- 项目是干什么的
- 前端/后端在哪里
- 临时文件放哪里
- 用什么命令启动、测试
- 哪些文件不能乱改
- 修改完成后怎么验收

这类信息如果不写，Hermes 只能靠猜。猜错了就会出现：文件放错目录、测试没跑、改了不该改的配置、临时文件堆在根目录。

```markdown

## 项目说明

这是一个用于_____的项目。
主要目标是：
- 
- 
- 

## 目录规则

- 正式源码放在：
- 文档资料放在：
- 临时实验放在：
- 测试文件放在：
- 不要把临时文件、截图、压缩包直接放在项目根目录。

## 常用命令

安装依赖：
启动开发环境：
运行测试：
代码检查：
构建项目：

## 工作规则

开始修改前，先阅读相关文件，不要直接猜项目结构。
修改代码后，尽量运行对应测试或检查命令。
如果无法运行测试，需要说明原因。
不要主动重构无关模块。
不要删除用户已有内容，除非用户明确要求。

## 输出要求

完成任务后，请说明：
- 修改了哪些文件
- 解决了什么问题
- 是否运行了测试
- 还有哪些风险或后续建议

## 禁止事项

不要把 API Key、Token、密码、私钥写进项目文件。
不要把真实密钥写进 `SOUL.md` 或 `AGENTS.md`。
不要在未确认的情况下执行删除、覆盖、迁移、发布等高风险操作。
```

> 大型项目可在 `frontend/`、`backend/` 等子目录里放更细的 `AGENTS.md`。

## 四、使用顺序

1. **先改 `SOUL.md`** — 让 Hermes 知道你的长期偏好（结论先行、讨厌废话、主动核实等）
2. **放项目 `AGENTS.md`** — 在每个重要项目根目录放一份，写明项目的目录、命令、测试和禁区
3. **在项目根目录启动 Hermes** — 这样它更容易先读到项目级规则

## 五、不要写得太满

- `SOUL.md` 控制在 4-8 条核心偏好
- `AGENTS.md` 先写清楚项目结构、常用命令、测试方式和禁止事项
- 先让它有一个能稳定工作的底座，用一段时间后再慢慢补

规则文件不是一次写完的——像 README 一样，随使用不断更新。

## 六、常见错误

1. **把 `SOUL.md` 放到项目里** — `SOUL.md` 是全局身份文件，默认在 `~/.hermes/SOUL.md`
2. **把项目命令写进 `SOUL.md`** — 端口、启动命令、目录结构应放进 `AGENTS.md`
3. **规则写得太空** — "请认真工作"对 Agent 没用，要写清楚文件放哪、什么命令、什么不能碰
4. **把密钥写进去** — 不要把 API Key、Token、密码写进规则文件

## 七、核心原则

> 工具越强，越需要规则。这就是 Agent 真正好用的开始。

Hermes 的重点不只是"能聊天"。真正有价值的地方，是能在规则、记忆和技能之间慢慢形成稳定工作方式。前提是，你要先把规则写清楚。
