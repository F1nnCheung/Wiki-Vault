# Agentic RAG 四种模式

传统 RAG 的执行路径是固定的：用户提问 → Embedding → 向量检索(top-K) → 塞进 Prompt → 生成答案。这是一种「一次性赌注」——第一次检索结果质量差，没有任何机制能发现并重试。根据多个生产系统的统计，**15%-30% 的 RAG 失败**源于检索质量问题，而这些失败在传统架构里无从发现，更无法修复。

Agentic RAG 的核心思路：把检索这件事交给 Agent 来决策。要不要检索？检哪个数据源？检完发现不够怎么办？这些都由 Agent 在运行时自主判断，而不是在代码里硬编码。

## 传统 RAG 的三个死穴

| 死穴 | 描述 |
|------|------|
| **单次检索，没有回头路** | 第一次检索结果质量差，没有任何机制能发现并重试。系统会用一堆不相关的 chunk 生成一个"看起来很正经"的错误答案 |
| **单一数据源，知识孤岛** | 用户的问题往往跨越多个数据域——文档库、数据库、实时 API。传统 RAG 只能连一个向量库 |
| **无法分解复杂问题** | 「对比 Q3 和 Q4 的用户留存率，结合客服反馈分析原因」——需要至少三次独立检索再汇总，传统管道只能硬塞进一个 query |

## Agentic RAG vs 传统 RAG

| 能力维度 | 传统 RAG | Agentic RAG |
|----------|----------|-------------|
| 检索策略 | 固定（单次向量搜索） | 动态（Agent 选数据源、改查询） |
| 检索轮次 | 永远是 1 次 | 按需决定（1 到 N 次） |
| 质量评估 | 无 | Agent 给检索结果打分 |
| 错误恢复 | 无 | 检测到失败后换策略重试 |
| 工具调用 | 无 | 可调用 API、SQL、网页搜索 |
| 查询分解 | 无 | 把复杂问题拆成子问题 |

## 四种模式全景

Agentic RAG 不是一种固定架构，而是四种可组合的模式。

### 模式一：路由型 RAG（Routing RAG）

**核心逻辑**：Agent 先理解问题意图，再决定去哪个数据源取数据。

- 知识分散在多个后端：向量库、SQL 数据库、知识图谱、实时 API
- 用户问「Q3 营收」→ 走 SQL；问「产品规格」→ 走向量库；问「最新股价」→ 走实时 API
- 关键实现：将每个数据源封装成 Tool，让 LLM 自主选择调用哪个

**适用场景**：多数据源场景（企业内部知识库 + 数据库 + 实时 API）

### 模式二：多步型 RAG（Multi-step RAG）

**核心逻辑**：Agent 把复杂问题拆解成子问题，依次检索，最后汇总。

- 单个 Query 需要多轮独立检索才能回答
- 例如「新定价方案上线后流失率怎么变化，客服反馈如何」→ 拆成三个独立子查询，分别检索，最后合并推理
- 每步子查询的结果作为下一步的上下文输入

**适用场景**：需要跨文档综合推理的复杂问题（对比分析、因果推断）

### 模式三：纠错型 RAG（Corrective RAG / CRAG）

**核心逻辑**：检索 → 评分 → 相关就生成，不相关就改写 Query 重试，完全没有就降级到网页搜索。

CRAG 是**工程价值最高**的模式，工作流程：

```
检索 → 评分（LLM 判断相关性）
  ├── relevant → 直接生成
  ├── irrelevant → 接受并生成（有部分信息）
  └── none → 改写 Query → 重新检索
                └── 超过重试上限 → 降级生成
```

**LangGraph 实现要点**：
- State 中维护 `retrieval_grade`（relevant/irrelevant/none）和 `retry_count`
- 评分节点用 `withStructuredOutput` 让 LLM 输出结构化评分
- 条件路由：根据评分结果决定下一步（生成/改写/重新检索）
- 设置 `MAX_RETRY` 上限，防止无限循环

**适用场景**：实际项目里最实用的模式，适合大多数需要"能用"到"好用"的场景

### 模式四：自适应型 RAG（Adaptive RAG）

**核心逻辑**：Agent 在"要不要检索"这一步就做判断。

- Agent 先判断：这个问题是常识、上下文已覆盖，还是真的需要检索？
- 不需要检索就直接生成，避免无意义的检索开销
- 需要检索则选择合适的数据源和检索策略

**适用场景**：混合了简单常识问题和复杂知识问题的场景，避免简单问题也走完整检索流程

## A-RAG：中科大的层级检索接口方案

A-RAG 是中科大团队提出的 Agentic RAG 框架，作者定义了真正的 Agentic RAG 需要同时满足三个条件：

1. **自主策略选择**（Autonomous Strategy）：Agent 自己决定用什么检索工具
2. **迭代执行**（Iterative Execution）：可以多轮检索，不是一锤子买卖
3. **交错式工具调用**（Interleaved Tool Use）：检索和推理交替进行，而非先检索再推理

### 层级检索接口（Hierarchical Retrieval Interfaces）

A-RAG 给 Agent 暴露三个不同粒度的检索接口：

| 接口 | 粒度 | 描述 |
|------|------|------|
| `keyword_search` | 关键词级 | 精确匹配专有名词、型号、人名。不建倒排索引，查询时直接对语料做文本匹配 |
| `semantic_search` | 句子级 | 向量余弦相似度匹配，返回语义最相关的句子片段 |
| `chunk_read` | 文档块级 | 读取完整文档块。前两个工具只返回片段摘要，Agent 觉得某个 chunk 值得深入看时才调用 |

没有预定义流程，没有固定顺序。Agent 自己决定什么时候用哪个工具、用几次、什么时候停。三个工具覆盖了从关键词级、句子级到文档块级的三层信息粒度——Agent 可以先粗筛再精读，也可以直接精确命中。

## NaviRAG：清华的"主动导航"方案

NaviRAG 是清华大学提出的方案，核心创新是让 RAG 在长文档问答中学会**主动导航**——不是被动地检索 chunk，而是像人类阅读长文一样主动规划"先读哪里、再读哪里、什么时候该停"。

- 在长文问答（Long-Document QA）任务上，F1 分数提升 4.8 个百分点
- 核心机制：Agent 维护一个"阅读状态"，根据已读内容动态决定下一次检索的焦点
- 相比传统方案（固定窗口滑动或一次性全文检索），NaviRAG 的检索效率更高、答案更聚焦

## 实践建议

### CRAG 落地清单

1. **评分节点不要只返回"相关/不相关"**，至少三档：relevant（高度相关）/ irrelevant（有部分信息）/ none（完全无关）
2. **设置重试上限**（建议 MAX_RETRY=2），防止改写-检索循环跑死
3. **保留降级路径**：超过重试上限仍然生成答案，不要直接报错
4. **评分节点用便宜模型**（如 GPT-4o-mini），生成节点用强模型
5. **记录每次评分的 reason**，方便后续分析检索质量问题

### 从预定义工作流到 Agent 决策的演进路径

1. **第一阶段**：传统 RAG，固定流程，先跑通
2. **第二阶段**：加入混合检索（BM25 + 向量），提升召回质量
3. **第三阶段**：引入 CRAG，加入检索评分和改写重试
4. **第四阶段**：如果多数据源，引入路由型 RAG
5. **第五阶段**：如果复杂问题多，引入多步推理

不要一步到位做全功能 Agentic RAG——在 CRAG 阶段通常就能解决 80% 的问题。

## 总结

- 传统 RAG 是"一次性赌注"，15%-30% 的失败源于检索质量问题且无法自我修复
- Agentic RAG 四种模式可组合：路由型（选数据源）、多步型（拆解问题）、CRAG（纠错重试）、自适应型（按需检索）
- CRAG 是工程价值最高的模式：检索 → 评分 → 改写 → 重试，代码量不大但效果立竿见影
- A-RAG 提出了真正的 Agentic RAG 三条件：自主策略选择 + 迭代执行 + 交错工具调用
- 实践中建议从 CRAG 开始，逐步叠加其他模式
