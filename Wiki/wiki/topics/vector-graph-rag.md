# Vector Graph RAG

传统解决 RAG 多跳问答（Multi-hop QA）的标准答案是知识图谱 + 图数据库。但你需要提取三元组、部署 Neo4j、学 Cypher 查询语言、运维向量库 + 图库两套系统——系统复杂度翻倍，运维成本和学习门槛双双拉满。

Vector Graph RAG 提出了一种新思路：**不引入图数据库，只用向量数据库（Milvus）解决 RAG 的多跳困境**。

## RAG 的多跳困境

普通 RAG 对付单跳问题——比如「爱因斯坦哪年出生」「Milvus 支持哪些索引」——向量一搜就找到了。但只要问题拐个弯、藏个隐含关系，普通 RAG 直接失效。

例如：「糖尿病的一线用药有哪些需要注意的副作用？」

这个问题需要两步推理：先知道糖尿病的一线用药是二甲双胍，再去查二甲双胍的副作用。Naive RAG 的向量搜索大概率只会暴力匹配"糖尿病"、"副作用"相关的段落，但它没法自己找到**二甲双胍**这个关键中间环节。

这就是多跳推理的典型死穴：答案散在不同段落，要靠实体关系把它们串成一条链，而纯语义搜索抓不住实体间的隐藏关联。

## 核心思路：用向量数据库构建逻辑图结构

知识图谱里的"关系"，说到底就是一段文本。既然文本能转向量、能被 Milvus 检索，**关系当然也可以**。

例如 `(二甲双胍, 是一线用药, 2型糖尿病)` 这条关系，在图数据库里是一条有向边。但换个角度看，它就是一句话：「二甲双胍是 2 型糖尿病的一线用药」——完全可以做 Embedding，存到 Milvus 里。

### 数据结构：三个 Collection 构建逻辑图

在 Milvus 中建三个 Collection，通过 ID 互相引用：

| Collection | 存储内容 | 作用 |
|------------|----------|------|
| **实体 Collection** | 每个实体的名称 + 描述 + Embedding | 语义检索实体 |
| **关系 Collection** | 每条三元组的文本表示 + Embedding | 语义检索关系 |
| **段落 Collection** | 原始文档段落 + Embedding | 最终答案的上下文来源 |

三个 Collection 之间通过实体 ID 和关系 ID 互相引用，构成一个**逻辑上的图结构**。没有图数据库，但图遍历的能力一样不落。

## 检索流程：子图扩展代替图遍历

### 传统图 RAG 的做法

```
Query → 实体链接 → Neo4j 图遍历（1-hop, 2-hop...）→ 收集路径 → LLM 生成
```

### Vector Graph RAG 的做法

```
Query → Embedding → 向量检索实体（找起点）
  → 向量检索关系（找边）
  → 根据关系找到关联实体（子图扩展）
  → 向量检索相关段落（找上下文）
  → 一次 LLM 重排 → 生成答案
```

**关键差异**：用子图扩展代替图遍历，用一次 LLM 重排代替多轮 Agent 循环。多跳问答不需要多次检索-生成循环。

## 性能表现

实测数据（三大多跳问答基准）：

- 平均 Recall@5 达到 **87.8%**，直接超过 HippoRAG 2
- 全程只需要**一个 Milvus 向量数据库**

## 适用场景 vs 局限

### 适用场景

- 已有 Milvus 向量数据库基础设施，不想引入图数据库
- 多跳问答的跳数在 2-3 跳以内
- 实体关系相对明确（人物/产品/组织/事件）
- 团队不熟悉 Cypher/图数据库运维

### 局限性

- 深层推理（4+ 跳）性能不如原生图数据库
- 关系复杂且稠密的场景（如社交网络分析）不适合
- 需要高质量的关系抽取预处理
- 作为研究方案仍处于早期阶段

## 与 Graph RAG（Microsoft）的对比

| 维度 | Graph RAG（Microsoft） | Vector Graph RAG |
|------|------------------------|-------------------|
| 图存储 | Neo4j / 图数据库 | Milvus 向量数据库 |
| 核心机制 | 社区摘要 + 全局搜索 | 子图扩展 + 向量检索 |
| 适用问题 | 全局性问题（「这个数据集的主要主题是什么」） | 多跳推理问题（「A 和 B 之间有什么关系」） |
| 基础设施 | 向量库 + 图库，两套系统 | 单一 Milvus 向量库 |
| 成熟度 | 微软出品，文档完善 | 较新的开源方案 |

## 总结

- Vector Graph RAG 证明了：**在向量数据库里也能做多跳推理**，不一定需要图数据库
- 核心思路：把实体、关系和段落都存为向量，通过 Collection 间的 ID 引用构建逻辑图
- 用子图扩展代替图遍历，用一次 LLM 重排代替多轮 Agent 循环
- 实测 Recall@5 达 87.8%，超过 HippoRAG 2，且只需要一个 Milvus
- 适合 2-3 跳推理场景，深层推理和关系稠密场景仍需要原生图数据库
