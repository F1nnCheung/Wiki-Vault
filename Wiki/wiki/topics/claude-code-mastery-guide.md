# Claude Code 从入门到精通（花叔 v2.0.0）

> 花叔的 Claude Code 从入门到精通 v2.0.0 是一份 10 章约 75 页的完整指南，涵盖从安装到多 Agent 协作的全流程。核心洞见：**工具会过时，功能会更新，但好的思维方式不会贬值。**

## 三层模型：你的时间该花在哪

花叔提出 Claude Code 的三层能力模型，这是全书最重要的思维框架：

```
Prompt 层        → 你说的话（每次对话都要重新投入 → 一次性回报）
    ↓
Context 层       → AI 能看到的信息（写一次 CLAUDE.md，持续生效 → 复利回报）
    ↓
Harness 层       → 自动化环境（搭一次流程，永久运行 → 指数回报）
```

- **Prompt 层**：你在终端里输入的每一句话。大多数初学者的全部交互都停留在此
- **Context 层**：Claude 在回答前已经"看到"的所有信息——CLAUDE.md、项目文件结构、git 历史、package.json
- **Harness 层**：你构建的自动化环境——Skills 封装工作流、Hooks 自动触发、MCP 连接外部服务、Agent Teams 并行协作

> 一个比喻：Prompt 是你开口说话，Context 是你提前准备好的 PPT，Harness 是你搭建的整个舞台。观众（Claude）的表现，取决于这三层的综合质量。

**核心原则**：初学者把所有精力花在 Prompt 层，反复琢磨措辞。高手的做法是——尽量把信息沉淀到 Context 层，把重复劳动交给 Harness 层，只在 Prompt 层处理真正需要临时决策的事情。

**如果只记住一件事**：把时间花在构建 Context 和 Harness 上，而不是优化 Prompt。

## CLAUDE.md：给 AI 一张地图

### 为什么它是最重要的文件

Claude Code 每次启动新会话，第一件事就是读 CLAUDE.md。项目结构、代码风格、测试命令、常见陷阱，Claude 都从这里了解。Shrivu Shankar（Abnormal AI 的 AI 战略 VP）称它为 agent 的"**宪法**"——短、原则性强、不处理细节。

### 护栏，不是手册（Guardrails, Not Manuals）

新手最常犯的错：试图把每个函数的用法、每个文件的作用、每个 API 的参数都塞进去。写了几千行，Claude 光读这个文件就吃掉大量上下文。

**Boris（Claude Code 创建者）团队的 CLAUDE.md 只有约 2500 tokens（约 100 行）。**管理 Claude Code 这个产品本身的核心规则文件，就这么短。

正确的做法：**从护栏开始，基于 Claude 做错的事情来记录。**不要试图预先写一本完整手册，而是每次 Claude 犯错，就加一条规则。

### 该写什么、不该写什么

| 该写 | 不该写 |
|---|---|
| Claude 猜不到的 Bash 命令（自定义构建脚本） | Claude 读代码就能知道的事（"这是一个 React 项目"） |
| 与默认不同的代码风格偏好 | 标准语言规范（Claude 已经知道） |
| 测试命令和偏好的测试框架 | 详细 API 文档（给链接，不要全文粘贴） |
| 项目架构决策和背景 | 频繁变化的信息 |
| 开发环境的坑（特殊的环境变量） | 文件逐一描述（Claude 会自己看文件树） |
| 常见陷阱和修复方式 | "写整洁代码""遵循最佳实践"这种废话 |

### Shrivu 的几个反模式补充

- 不要用 `@` 引用大文档——会被完整嵌入上下文。正确做法是提到路径，告诉 Claude 什么情况下去读
- 不要只写"永远不要做 X"——当 Claude 觉得必须做 X 时它会卡住。永远提供替代方案
- 如果某个 CLI 命令复杂到需要在 CLAUDE.md 里写几段话解释，那说明这个命令本身需要简化

### 层级结构（Hierarchy）

CLAUDE.md 是一套层级系统，Claude Code 自动按顺序读取：

```
~/.claude/CLAUDE.md     ← 全局级：所有项目共用的个人偏好
./CLAUDE.md             ← 项目级：检入 git，与团队共享
./src/CLAUDE.md         ← 子目录级：monorepo 中特定模块的规则
./src/api/CLAUDE.md     ← 更深层子目录
```

### 迭代飞轮：越用越好的系统

```
第一周：空文件 → 第二周：护栏初现 → 第三周：错误骤降 → 稳定后：微小维护
```

Mitchell Hashimoto（HashiCorp 联合创始人）描述过一模一样的过程：配置文件里的每一行都对应 agent 过去犯过的一次错。文件是活的，一直在长。

Boris 的原话：**"Claude 非常擅长为自己编写规则。"**你告诉它犯了什么错，它自己就能写出精确的规则防止下次再犯。

## Auto 模式：更安全的自动驾驶

Anthropic 内部数据显示：**93% 的权限请求被用户直接批准了。**审批疲劳让安全机制形同虚设。Auto 模式用一个 AI 分类器替你做权限判断：安全操作自动放行，危险操作才拦截。

### 两层防御

1. **输入层**：Prompt Injection 探测器扫描 Claude 读取的所有内容，发现可疑内容时附加警告
2. **输出层**：Transcript 分类器在每个操作执行前评估风险
   - 第一阶段：快速判断（一个 token 的 yes/no），8.5% 误报率
   - 第二阶段：只对可疑操作做深度推理，把误报率降到 0.4%

### Auto 模式 vs --dangerously-skip-permissions

| 维度 | Auto 模式 | --dangerously-skip-permissions |
|---|---|---|
| 安全性 | 有 AI 分类器评估每个操作 | 完全无保护 |
| 危险操作 | 会被拦截 | 直接执行，无任何提示 |
| Prompt Injection 防护 | 有输入层探测器 | 无 |
| 适用场景 | 日常开发 | 完全隔离的沙箱环境、CI/CD |

### 三层权限体系

| 方式 | 省心程度 | 安全程度 | 适合谁 |
|---|---|---|---|
| Auto 模式 | 高 | 中 | 大多数日常开发者 |
| /permissions 白名单 | 中 | 高 | 团队使用、需要精细控制 |
| 逐个确认（默认） | 低 | 最高 | 高风险操作、初学阶段 |

## Computer Use：AI 长了眼睛和手

Claude Code 的 Computer Use 功能让 Claude 直接看到你的屏幕截图，然后操控鼠标和键盘——不是模拟，不是调 API，是真的在看你的屏幕、移动你的光标、点击你的按钮。

### 最顺手的场景

- **测试 Web 应用的 UI**：像用户一样点击页面、填表单、看到渲染结果，发现视觉问题
- **操作没有 API 的桌面软件**：老旧的管理后台、只有 GUI 的工具
- **自动化重复的 GUI 操作**：批量处理文件、在多个窗口之间来回复制数据
- **调试 Chrome 扩展**：popup、content script 效果只能在浏览器里看到

### 为什么这件事重要

Computer Use 打破了 AI 只能操作"文本世界"的边界。**AI 的操作范围从"能写代码的地方"扩展到"屏幕上看得到的一切"。**一个 PM 可以对 Claude 说"帮我在 Figma 里把这个按钮改成蓝色"，一个运营可以说"帮我在后台把这批用户状态改成 VIP"，不需要理解任何技术概念。

### 当前限制

- 慢：每一步都需要截图→分析→决定→执行
- 精细操作不靠谱：拖拽滑块到精确位置、在密集表格中选中特定单元格
- 不适合需要快速反应的场景：动画、实时交互、游戏测试

> 现阶段最好的定位：把它当成一个耐心但手速慢的测试员。给它"按照固定流程重复操作"的任务，它做得很好。

## Voice Mode：开口说话就能编程

按住空格说话，松开发送。`/voice` 进入语音模式，支持 20 种语言。

### 语音比打字好的场景

- **手不方便的时候**：走路想到修复思路、做饭时想起需求
- **脑暴的时候**：想法哗哗地冒，打字速度跟不上大脑
- **描述空间和视觉概念的时候**："我想要一个左边是侧边栏、右边分上下两栏……"

### 交互方式变化的深层意义

键盘→鼠标→触屏→语音。每次交互方式变了，用工具的人就多了一大圈。现在把 Voice Mode 和 Computer Use 放在一起看：**Voice 说需求，Computer Use 执行操作。**人可以完全脱离键盘和代码，纯靠说话让 AI 帮你构建东西。

> 一个实用组合：语音快速启动任务，然后切回键盘输入精确的细节和约束条件。

## Agent Teams 与多 Session 协作

### Subagents：给主 Session 叫个帮手

在 `.claude/agents/` 目录下放一个 `.md` 文件就定义了一个 subagent。每个 agent 可以定义自定义名称、工具集权限、权限模式，甚至指定使用的模型。

**Subagents 最重要的特性不是"专业分工"，而是独立上下文。**每个 subagent 运行在自己的上下文窗口中，不消耗主 session 的上下文空间。

推荐组合：security-reviewer（涉及认证/权限时自动调用）+ verify-app（修改完成后自动验证功能）。

### Agent Teams：让它们自己协调

多个 session 之间能互相通信、协调分工。核心理念：**不是你协调多个 agent，而是让 agent 自己协调。**

**Writer/Reviewer 模式**：一个写代码、另一个审代码。Writer 实现功能，Reviewer 指出问题，形成迭代循环。比单个 agent 好不少——和人类团队一样：写代码的人容易陷入自己的思路，审代码的人能从不同角度发现问题。

**Coordinator Mode**：复杂任务自动走四阶段：
1. Research：多个 worker 并行调查代码库
2. Synthesis：coordinator 综合发现生成规格说明
3. Implementation：worker 按规格做精准修改
4. Verification：验证结果

### Fan-out 批处理：人海战术的 AI 版

同一个操作需要对大量文件重复执行时，用 `/batch` 命令或 shell 循环 + `claude -p` 非交互模式，启动数十个 Claude 实例并行处理。一个人加 Claude，抵得上一个工程团队花一周做的迁移工作。

### 异步工作：不一定要盯着电脑

- **Remote Control**：生成连接链接，手机上远程创建和管理本地 session
- **`/schedule`**：云端定时任务，电脑关机了照样按时跑（依赖更新、安全扫描、日报生成）
- **`/loop`**：本地长时间运行，最多无人值守 3 天

> 把 AI 当成"夜班团队"：白天你定方向做决策，晚上它执行。

## 六个高频反模式（Anti-Patterns）

| 坑 | 表现 | 解决方案 |
|---|---|---|
| **坑1：一个会话什么都塞** | 修 bug、加功能、重构、写文档全在一个会话 | 一个会话聚焦一个任务，做完就 `/clear` |
| **坑2：反复纠正越改越偏** | 纠正两次不行，花在纠正上的时间比自己动手还多 | 纠正两次不行果断 `/clear` 重来，重新描述需求 |
| **坑3：看着像对的就接受了** | 输出看着合理，没实际跑一下，过几天发现 bug | 每一轮改动都实际运行一次 |
| **坑4：过度微操** | 每写一个文件都要看，每改一行都要评论 | 关注结果，让 Claude 把完整任务做完再看 |
| **坑5：需求模糊怪 AI 不懂你** | "帮我优化一下""让页面好看点" | 给具体的、可验证的需求 |
| **坑6：不写 CLAUDE.md** | 每次新会话都要重新解释项目背景 | 翻到 CLAUDE.md 章节反复看 |

## Claude Code 内部架构：TAOR 循环

Claude Code 的心脏是一个叫 **TAOR** 的 Agent 循环：

```
Think → Act → Observe → Repeat
  ↑                           ↓
  └────── 未完成则继续 ────────┘
```

先思考当前状态，决定下一步做什么；然后调用一个工具执行操作；观察返回的结果，判断任务是否完成；没完成就继续循环。整个过程可能转几十圈才停下来。

这解释了为什么 Claude 有时候看起来"绕弯路"——它是一个不断试探和调整的循环体，每一步都在根据新的观察做决策。**给 Claude 明确的验证标准特别重要**：循环需要一个停止条件。如果需求描述模糊，它不知道什么时候算"做完了"，就会不停地循环下去。

### 40+ 工具归结为 4 个能力原语

所有能力归结为：**Read**（读文件/日志/网页）、**Write**（创建/编辑文件）、**Bash**（执行命令/脚本/测试）、**Web**（浏览网页/调 API）。

### 技术栈趣闻

Claude Code 运行在 **Bun** 上（不是 Node.js），用 React 的 **Ink** 框架来渲染终端 UI，全部用严格模式 TypeScript 编写，Schema 验证用 Zod。这就是为什么它能有那么丰富的终端交互体验——权限确认弹窗、多行代码高亮、进度指示器，这些用 React 的组件模型就很自然。

## 产品开发的实战教训

### 花叔的核心原则

1. **先 SPEC 再代码**：没有规格说明不动手，SPEC.md 是需求基准线
2. **AI 写代码，你做决策**：技术选型、架构方向、功能取舍永远自己拍板
3. **验证每步、不累积问题**：Claude Code 一小时能写 2000 行，不验证问题会以更大成本爆发
4. **不要在一个 session 里做太多不相关的事**：前端/后端/部署/修 bug 分 session
5. **产品感知才是最大杠杆**：AI 能让执行速度提升 10 倍，但方向错了，只是以 10 倍速度走向错误

### 产品开发常见陷阱

| 陷阱 | 表现 | 解决方案 |
|---|---|---|
| 需求膨胀 | 做着做着不断加功能，永远做不完 | 回 SPEC.md，不在规格内的功能记 todo |
| 上下文污染 | session 越来越长，Claude 忘记代码结构 | 及时开新 session，让 CLAUDE.md 承载上下文 |
| 不验证就继续 | 连续写 10 个文件，发现第 2 个就有 bug | 每完成一个模块必须验证 |
| 环境变量混乱 | 本地能跑，部署后各种 undefined | CLAUDE.md 列出所有环境变量，部署前 checklist |
| 过度依赖 AI 判断 | AI 说"这个方案最好"就采纳 | AI 给方案，你做决策，架构选型永远自己拍板 |
