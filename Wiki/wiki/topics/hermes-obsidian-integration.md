# Hermes + Obsidian 知识库集成

**核心方案**：Obsidian 做存储底座，Hermes Agent 做执行层。让知识库从「仓库」变成「生产线」。

> 问题不在 Obsidian。Obsidian 的强项是存储、链接、沉淀。但它天然不会主动推流程——不会帮你判断一篇资料值不值得收，不会把资料拆成选题，也不会把一个选题推成草稿。

## 一、核心认知：知识库最大的浪费不是「不够大」，而是「存了不用」

很多人一听「AI + 知识库」就往向量数据库、RAG、知识图谱上想。实际应用中，核心就盯一件事：

```
资料进来 → 判断价值 → 拆成选题 → 生成草稿 → 发布归档
```

对内容创作者来说，知识库最重要的不是「以后搜得到」，而是「现在能不能用起来」。

- 一篇收藏没进过选题池 = 死了
- 一个灵感没变成过大纲 = 心理安慰

## 二、目录设计：不是给人看的，是给 Agent 认路的

```
📁 01-Sources     ← 外部资料入库
📁 02-Accounts    ← 各账号的内容资产
    └─ 超级猛
       ├─ Topics    ← 选题池
       ├─ Drafts    ← 草稿区
       └─ Published ← 发布归档
📁 03-Frameworks  ← 内容框架、写作规范、账号定位
```

以前搭目录是「方便自己以后找」。现在多了一重考虑：**Hermes 能不能读懂？能不能根据目录判断一个文件该进 Source、Topic、Draft 还是 Published？**

最小流转链路：

```
Source Note → Topic Note → Draft Note → Published Note
```

只要这条线跑顺了，知识库就已经从「仓库」变成了「生产线」。

## 三、实操流程

### 第一步：入库判断

把链接或正文扔给 Hermes 走第一轮，不只是收藏，而是先过一遍「编辑台」：

> 判断这篇内容是否值得收入 Obsidian 内容中台。如果值得，生成 Source Note，包含：
> 1. 核心摘要
> 2. 关键观点
> 3. 适合哪个账号
> 4. 可延展成哪些选题
> 5. 建议放入的目录和文件名

**价值**：Hermes 会替你做初筛——这玩意儿值不值得继续加工。知识库里最耗精力的不是整理，而是低质量资料太多。第一步不是多存，是让 Agent 帮你挡掉一批不值得推进的东西。

### 第二步：拆选题

> 基于这篇 Source Note，为「超级猛」账号拆 3 个选题。每个选题包含：
> 1. 一句话定义
> 2. 目标读者
> 3. 用户痛点
> 4. 核心承诺
> 5. 标题候选（3 个）
> 6. 简短提纲
>
> 判断哪个最适合近期发布。

同一篇资料可以写成工具测评、方法论、踩坑总结。角度不定，后面越写越散。Topic Note 的作用就是先把「我要写什么」锚住。

选题四标准：
- 有明确的用户痛点
- 有真实案例可引用
- 能讲成可复现的工作流
- 能给出可操作的方法

### 第三步：生成草稿（但别让 AI 直接定稿）

> 基于这个 Topic Note，写一版适合「超级猛」账号的长文草稿。要求：
> 1. 从我的使用场景和判断出发，不要写成工具说明书
> 2. 结构清楚，但别像模板
> 3. 每节都有具体做法
> 4. 少用空泛形容词，多写真实问题和取舍
> 5. 结尾给出下一步行动建议

> ⚠️ **重要坑**：很多 AI 工具文容易写成「某工具是什么、有什么功能、有什么优势、适合谁」。看着很完整，读起来像产品介绍页。真正有用的写法是从一个具体问题切入：为什么需要它？之前卡在哪？接进来之后哪一步变了？

AI 产出只当初稿用，人工过两轮：
- **第一轮**：删掉正确但没信息量的话
- **第二轮**：补上真实判断和使用细节

AI 帮你铺结构，但文章有没有可信度，最后看作者有没有自己的判断。

### 第四步：把固定流程写成 Skill

把稳定流程沉淀成 Skill，不用每次都重新教：
- Source Note 的生成规范
- Topic Note 的字段模板
- 账号定位、选题偏好、表达风格
- 公众号 vs 小红书的差异化要求

规则一旦沉淀下来，Agent 才会越来越像工作流成员，而不是临时聊天窗口。

### 第五步：发布后归档

每篇发布后补一条归档记录：

```yaml
标题: 
发布时间: 2026-04-28
平台: 公众号
对应Topic: [[Topics/xxx]]
对应Draft: [[Drafts/xxx]]
链接: 
封面文案: 
数据复盘:
```

这一步的价值不在「记录」，在**反馈**。全链路应为：

```
Source → Topic → Draft → Published → Review → New Topic
```

这才叫「内容中台」——不是一个资料库，是一个能持续产生判断的系统。

## 四、适合谁

如果你中了三条以上，值得试试这套方法：
- 收藏了大量文章但很少转成输出
- Obsidian 目录越来越复杂，打开频率越来越低
- 同时管多个账号，经常不知道素材该分给谁
- 想让 AI 不只是一次性写作工具，而是参与整个流程管理
- 已经在用 Claude Code、Cursor、Codex 或 Hermes

## 五、最小启动版本

不用一步到位，先搭四个目录：

```
📁 01-Sources
📁 02-Topics
📁 03-Drafts
📁 04-Published
```

固定三个提示词：
1. **入库判断**：「请把这篇资料整理成 Source Note，判断是否值得继续推进。」
2. **拆选题**：「基于这篇 Source Note，为我的账号拆 3 个可写选题，推荐最适合近期发布的一个。」
3. **出草稿**：「基于这个 Topic Note，写一篇有真实使用场景、有方法步骤、有个人判断的草稿。」

三步跑顺后，知识库就不再是你一个人手动维护的仓库，而是开始变成一条有人帮你推着走的生产线。
