# RAG 优化 20 法：从"能跑"到"能用"

> 一个下午就能搭出能跑的 RAG 原型——但搜出来的东西牛头不对马嘴、明明库里有的内容就是找不到、问得稍抽象一点就开始编。RAG 的本质不是"让 AI 更聪明"，而是**让正确的信息在对的时机出现在对的位置**。本文按请求生命周期拆解 20 种优化方案。

## 全流程总览

```
数据入库 → 检索前（查询优化） → 检索阶段 → 检索后（结果处理） → 生成阶段
```

**每一个阶段的改进，都在给下一个阶段"减负"。**

---

## 一、数据入库：地基打不好后面全白费

### 1. 语义分块：别再按字符数切了

按 500 字固定长度切块就像把一本书每 10 页拆下来订成一册——刚好把完整论证拦腰截断。**语义分块**按句子、段落或 Markdown 标题来切，保证每块是一个完整的逻辑单元。

### 2. 小块检索 + 大块投喂

文档切成**很小片段**做向量检索（小块精度高），命中后把所在"整段大哥"调出来喂给 LLM。就像图书索引：你用关键词查到页码，但真正读的是那一整页内容。

### 3. 元数据过滤：先粗筛再细搜

存文档时顺手存上时间、作者、类型。检索时先 `WHERE` 一下（"只要 2024 年以后的""只要 Python 相关的"），再做语义匹配。一个简单的条件过滤就能把搜索范围从百万级缩到千级。

### 4. 摘要索引：让宏观问题先找到大方向

长文档被切碎后，用户问"这本书讲了什么"时任何一个片段都答不上来。入库前让 LLM 给每个长文档生成全局摘要，单独向量化。宏观问题命中摘要 → 顺藤摸瓜找到细节。

### 5. Graph RAG：不只搜"内容"，还搜"关系"

把文档里的实体和关系抽取出来存进知识图谱。检索时**同时查向量库和图数据库**。想知道的不只是"这段话讲了什么"，还有"它跟别的东西是什么关系"。详见 [[../concepts/knowledge-graph|知识图谱]]。

### 6. 文档反向提问：提前帮文档"写考题"

入库前让 LLM 看着每段文本生成 5 个"如果有人要查这段内容，可能会怎么问"。把这些问题和原文绑在一起存入向量库。**相当于提前把所有可能的问法都做好了索引。**

---

## 二、检索前：用户的提问不是最优的查询词

用户提的问题往往口语化、简略、甚至有歧义。直接拿去搜等于让口语问题去碰书面语文档。

### 7. 查询重写：帮用户"翻译"成检索语言

调用 LLM 将用户的自然语言问题改写为更适合检索的标准问法。多了一步 LLM 调用，但省去了后续大量试错。**性价比极高。**

```
用户：今年一二月大概赚了多少钱，各项收入和纯利润分别是多少呀？
改写：今年1月至2月的净利润和各项收入明细是多少？
```

### 8. 多路查询：一个问题，五个角度

把用户问题扩展成 3-5 个不同角度的提问，分别检索，最后合并去重。**用广度换精度。**

### 9. HyDE：先"闭卷"写个假答案，再拿假答案去找真文档

这是最有意思的方法。让 LLM 先凭空写一个**假答案**，然后把假答案向量化去搜真正文档。因为**"答案和答案"在向量空间里的距离，往往比"问题和答案"更近。**用"假的"去找"真的"。

### 10. 查询路由：不是所有问题都该走向量检索

设计路由器根据意图分发：
- "今年第二季度销售额？" → **SQL 数据库**
- "这段代码为什么报错？" → **向量库**
- "你好，今天天气怎么样？" → **直接给 LLM**

**先分类，再处理。**

---

## 三、检索阶段：语义和关键词各有盲区

### 11. 稠密检索：懂语义，不懂术语

基于 Embedding 的语义搜索。搜"轿车"能找到"小汽车""私家车"。但遇到专业术语、缩写、代码函数名就懵了。

### 12. 稀疏检索：懂术语，不懂语义

基于 BM25 的关键词匹配。搜"OOM Killer"绝对不会漏掉"内存不足导致进程被杀"。但搜不到表达方式不同导致的关键词不匹配。

### 13. 混合检索 + RRF：两条腿走路

把稠密和稀疏结果用 **RRF（倒数排名融合）** 算法合并。两种方法的盲区刚好互补。**语义和关键词，一个都不能少。**

### 14. 微调嵌入模型：让你领域的话不再"听不懂"

通用 Embedding 模型对私有领域理解不好？用自己的数据微调一次。比如用内部代码文档训 BGE，它就能理解你项目中的特有函数名和术语。

---

## 四、检索后：最容易被忽视，但提升空间最大

很多人直接把 top_k 结果塞给 LLM。但检索出来的东西往往**有噪声、有重复、有废话**。

### 15. 重排（Reranking）：性价比最高的优化

向量检索用双塔模型（快但精度有限），重排用 Cross-Encoder（查询和文档拼在一起逐字比对，精度高但慢）。实践中只对初筛 top 20 做重排就够了。**召回对不代表排序对。**

### 16. 上下文压缩：帮 LLM 划重点

用 LLMLingua 等工具把检索段落里的废话剔除，留核心信息。**既省 Token，又降低 LLM "分心"的概率。**

### 17. MMR：搜出来的东西别全是同一句话

检索出 5 段文本，结果 4 段说的是同一回事——上下文窗口被白白浪费。MMR（最大边际相关性）保证结果**既相关又多样**。

---

## 五、生成阶段：有了好材料还要学会"好好说话"

### 18. Prompt 工程：一句话挡住大量幻觉

在 System Prompt 里加一句，效果立竿见影：

> "请严格基于提供的 Context 回答。如果 Context 中没有相关信息，请直接说'根据现有资料无法回答'，不要编造。"

**成本几乎为零，是所有优化里性价比最高的一项。**

### 19. 自我反思：答完之后回头看一眼

让 LLM 生成答案后再检查：自己说的内容是否确实来自给定上下文？如果没有，打回重写。在关键业务场景下，宁可多花几秒也别让幻觉毁掉用户信任。

### 20. 引用溯源：企业级 RAG 的标配

让模型在回答里标注来源 `[1]` `[2]`。当模型知道自己的每句话都要"对得上账"时，它会回答得更谨慎。

## 按场景选方案（组合 3-5 种）

| 场景 | 推荐组合 |
|------|---------|
| 通用知识问答 | 语义分块 → 混合检索 → 重排 |
| 企业私有文档 | 元数据过滤 → Graph RAG → 引用溯源 |
| 代码库问答 | 查询重写 → 稀疏检索 → 重排 |
| 长文档分析 | 摘要索引 → MMR → 上下文压缩 |

## 向量数据库选型：以 Milvus 为例

向量数据库是 RAG 的核心基础设施。以 Milvus 为例：

**基本概念**：Database（逻辑隔离）→ Collection（表）→ Field（列）+ Vector（浮点数组，768/1024/1536维）→ Entity（行）

### 索引类型速查

| 场景 | 索引 | 原理 |
|------|------|------|
| 小数据，绝对精准 | **FLAT** | 暴力全比对 |
| 常规百万级向量 | **IVF_FLAT** | 聚类分桶后查桶内 |
| 海量内存紧张 | **IVF_SQ8** | 向量压缩省内存 |
| 高并发低延迟 | **HNSW** | 图结构跳跃查找 |
| 平衡速度与精度 | **SCANN** | IVF+SQ8+HNSW 组合 |
| 海量SSD低成本 | **DISKANN** | 向量落盘，内存只留导航 |

**相似度量**：L2（欧式距离，越小越相似）、IP（内积，越大越相似）、COSINE（余弦，看方向不看长度）

## 嵌入模型选择

| 场景 | 类型 | 推荐模型 |
|------|------|---------|
| 纯文本语义检索 | 稠密 Dense | OpenAI `text-embedding-3-small`、Jina `jina-embeddings-v3` |
| 关键词精确命中 | 稀疏 Sparse/BM25 | BM25、TF-IDF、BGE-M3 sparse |
| 工业级中文 RAG | 混合 Hybrid | **BAAI/bge-m3**（一次调用同时输出 dense + sparse + colbert） |
| 多语言中文 | 多语言 Dense | bge-m3 / Jina / OpenAI |

> 参考 MTEB 排行榜（huggingface.co/spaces/mteb/leaderboard）：X 轴 = 参数量，Y 轴 = 平均得分。左边 = 低成本快速上线，右边 = 超长文档处理。

---

## 核心心法

1. **入库打好地基**：语义分块 > 字符分块；小块检索 + 大块投喂是黄金组合
2. **检索前帮用户"翻译"**：查询重写 + HyDE 假答案搭桥，缩小语义鸿沟
3. **检索时两条腿走路**：稠密 + 稀疏混合检索
4. **检索后别偷懒**：重排是性价比最高的优化
5. **生成时设好护栏**：Prompt 加一句"不知道就说不知道"，比什么花哨方案都管用

> 不是所有场景都需要 20 种全上。理解你的语义鸿沟最大在哪一环，选 3-5 种对症的组合，就足够让"能跑"的 RAG 变成"能用"的 RAG。
