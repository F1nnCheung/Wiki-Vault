# 混合检索

RAG 系统中，纯向量检索在语义匹配上很强大，但面对需要精确匹配的场景时有三个固有缺陷。混合检索的核心思路是：将**BM25 词法检索**与**稠密向量检索**并行执行，通过融合算法合并结果，让两种检索信号互补而非互斥。

## 纯向量检索的三个固有缺陷

### 词法精确性的丢失（Loss of Lexical Precision）

双编码器模型将所有 token 的表示聚合为一个固定向量。这种池化操作赋予了模型语义泛化能力，但代价是精确词法标识被抹掉了——池化在聚合语义的同时，不可逆地丢弃了 token 级的精确信息。当查询包含精确标识符（错误码、产品型号、领域术语）时，向量检索的"语义泛化"反而成了问题。

### 相关性评分的不可归因性（Non-attributability of Relevance Scores）

向量检索以余弦相似度为相关性分数，能反映向量在高维空间中的夹角，却无法像 BM25 那样将相关性归因到具体的词项匹配。无法回答"文档因匹配了查询中的哪个词而被判定为相关"。在需要审计追踪和合规验证的场景中，这种缺乏词项级归因路径的评分机制无法通过调整阈值或引入重排序来解决。

### 向量空间的静默漂移（Silent Embedding Drift）

嵌入模型迭代、知识库更新以及领域语言的变化，都会改变向量空间的结构。这种漂移通常表现为相关文档排名的缓慢下滑，很难察觉。当漂移累积到影响下游生成质量时，修复成本极高。

## BM25：词法精确性的回归

BM25 运行在倒排索引之上，进行精确 token 匹配——要么命中，要么不命中。它与向量检索形成互补：

| 特性 | BM25 | 向量检索 |
|------|------|----------|
| **确定性** | 相同查询在相同索引版本上产生相同结果，不依赖模型权重、随机种子或嵌入空间几何 | 受嵌入模型版本和权重影响 |
| **评分可追溯性** | 每个匹配结果可分解为：哪些词命中了哪些文档字段，各词的 IDF 权重与词频贡献 | 余弦相似度无法归因到具体词项 |
| **词法精确匹配** | 对缩写、产品代码、错误标识符、函数名等词法敏感的标识符处理准确 | 对精确字符串容易受语义模糊化影响 |
| **语义理解** | 弱（无法理解同义表达） | 强 |

## 检索策略选择

实际场景中检索策略的选择由语料特征、查询分布和系统约束共同决定。

### 纯向量检索适用场景

- 语料以自然语言叙述为主，领域术语、缩写和精确标识符密度低
- 用户查询以概念性问题为主
- 系统对可解释性无硬性要求
- 错误答案的业务代价低
- 处于快速验证阶段，需控制工程复杂度

> 例如：内容推荐、开放域知识问答。

### 混合检索适用场景

- 语料包含受控词汇密集的技术文档、法律合同、医疗记录或金融报告
- 用户查询类型混合，同时包含语义性问题和精确标识符查找
- 系统对准确性有硬性要求，错误答案会产生真实的业务或法律代价
- 需要审计追踪，系统必须能够说明检索决策的依据

## 混合检索的架构

### 融合策略

混合检索并行运行 BM25 词法检索和稠密向量检索，产生两个独立排序列表，再通过融合算法合并为单一排序结果。

#### 倒数排名融合（RRF）

将每个候选文档在各排序列表中的位次转化为分数：

```
RRF_score(d) = Σ (1 / (k + rank_i(d)))
```

- `k` 为约定常数（通常取 60）
- Elasticsearch 8.x、OpenSearch、Weaviate、Qdrant 等主流引擎原生支持

**优势**：对评分校准误差高度鲁棒。BM25 分数、余弦相似度乃至不同嵌入模型输出的相似度分布，在绝对数值上往往不可直接比较。RRF 通过在排名空间而非分数空间中进行融合，规避了跨检索器的分数校准难题。

**局限**：丢失评分幅度信息。相似度差异显著的文档，一旦处于同一排名位置，对 RRF 的贡献完全相同。

#### 凸组合（Convex Combination）

保留评分幅度，通过线性加权融合两种分数：

```
Score(d) = α × BM25_norm(d) + (1-α) × Vector_norm(d)
```

- α 直接控制词法检索的贡献权重，α 越大则词法匹配越占主导
- 在积累少量高质量标注查询-相关性对后，调优单一参数 α 能够在域内和域外评估上稳定超越 RRF
- 需要先对两种分数做归一化（min-max 或 z-score）

### 检索流程的外部化

OpenSearch 和 Elasticsearch 原生支持在单一请求内同时运行 BM25 和向量检索并合并结果。但内置实现将两种信号混合为单一分数，两个模式的贡献不可分离观测。

**外部化方案**：将 BM25 检索和向量检索作为独立步骤，在应用层实现融合：

- 每个模式的评分和命中情况可以独立观测
- 融合逻辑、权重调整和业务规则在常规代码中管理，不受引擎内部架构约束
- 支持自定义 RRF 权重、查询意图感知路由和多阶段检索链

**代价**：系统复杂度提高。对于无严格审计要求、处于快速验证的场景，一站式打包处理反而是最佳选择。

## 一种典型检索流水线

```
用户查询
  │
  ├──→ BM25 词法检索 ──→ 候选列表 A
  │
  ├──→ 向量语义检索 ──→ 候选列表 B
  │
  └──→ RRF / 凸组合融合
         │
         └──→ 重排序（Reranker）──→ Top-K 结果 → LLM 生成
```

## 与 Agentic RAG 的关系

传统的混合检索是人在代码里预先决定"用 BM25 + 向量，RRF 融合"。而 [[topics/agentic-rag-patterns|Agentic RAG]] 将检索策略的选择权交给 Agent：Agent 在运行时根据问题特征自主决定该用 BM25、向量检索还是两者都用，这被称为**自主策略选择**。

[[A-RAG]] 进一步提出了三层粒度的**层级检索接口**（Hierarchical Retrieval Interfaces）：
- `keyword_search`：关键词精确匹配
- `semantic_search`：语义向量检索
- `chunk_read`：读取完整文档块

Agent 自主决定什么时候用哪个工具、用几次、什么时候停——没有预定义流程，没有固定顺序。

## 总结

- 混合检索是两种检索机制的**互补**，而非简单叠加
- BM25 提供词法精确性和评分可追溯性，向量检索提供语义泛化能力
- RRF 适合快速验证（无需分数校准），凸组合适合深度优化（需要标注数据调 α）
- 外部化融合提供精细控制，内置融合降低系统复杂度——根据业务需求选择
- 混合检索是 Agentic RAG 中"自主策略选择"的基础能力之一
