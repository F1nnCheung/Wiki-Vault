# RAG 三种架构：Classic RAG、Graph RAG、Agentic RAG

RAG（Retrieval-Augmented Generation，检索增强生成）的核心思路：**先把相关资料找出来，再让大模型基于这些资料回答**——让模型从「凭印象回答」变为「带着资料回答」。但 RAG 发展至今已不只是「向量检索 + 大模型」这一种形态。

三个动词概括三种架构的本质差异：
- **Classic RAG**：检索（Retrieves）——找资料
- **Graph RAG**：连接（Connects）——找关系
- **Agentic RAG**：推理（Reasons）——决定下一步该查什么

---

## Classic RAG：先把相似内容找出来

### 核心流程

```
文档 → 切块（Chunk）→ Embedding → 向量数据库 → 用户查询时：
  用户问题 → Embedding → Top K 检索 → 拼入上下文 → LLM 生成答案
```

关键点：**不是只给问题做 embedding**，文档在入库时已提前 embedding，查询时实时 embedding。

### 适用场景

- FAQ 问答、政策查询、产品手册问答
- 客服知识库、内部文档检索
- 任何「答案就在某个段落里」的问题

### 优势
- 简单、快、成本低、工具链成熟
- 行为可预测，结果可解释

### 局限
- 擅长「找到相关内容」，不擅长「沿着关系继续往下找」
- 遇到跨文档、跨实体的多跳问题容易卡住
- 例如「A 的经理的经理是谁？」——答案不在一个段落，而在关系链中

> Classic RAG 解决的是「**资料在哪里**」的问题。

---

## Graph RAG：不只找文本，还要找关系

### 核心机制

在文本检索之外增加一层**知识图谱**结构。实体 = 节点（人/产品/零件/团队/供应商/系统），关系 = 边（汇报/使用/来自/负责/购买）。

```
文本线：切块 → Embedding → 向量库（保留）
图谱线：抽取实体 + 关系 → 构建知识图谱 → 沿图谱路径遍历查询
```

### 典型问题

用户问「哪些产品会受到半导体短缺影响？」：
- Classic RAG 可能找到「半导体短缺」相关文本片段
- Graph RAG 沿路径走下去：供应商 S → 芯片 Z → 电路板 Y → 产品 X，精准找出受影响范围

### 适用场景

- 影响分析（一个变化影响哪些下游对象？）
- 依赖分析（某组件被哪些对象依赖？）
- 组织关系查询、审批链查询
- 供应链分析、因果链解释

### 代价

- **建图成本高**：从文本中准确抽取实体和关系需要模型能力
- **维护成本高**：业务世界持续变化，图谱需持续更新
- **边界限制**：只能遍历已建模的节点和边，未建模的领域无路径可走

> Graph RAG 不是用来替代 Classic RAG 的，而是用来增强那些「关系密集型」问题的。

> Graph RAG 解决的是「**信息之间如何连接**」的问题。

---

## Agentic RAG：让系统决定下一步该查什么

### 核心机制

不再执行固定检索流程，而是让 **Agent 根据目标自主判断**：下一步该查什么、用什么工具、证据够不够、要不要重新规划。

```
问题 → Agent 拆解 → 选择工具/数据源 → 执行查询 → 评估结果 → 
  证据不足？→ 换工具/换数据源 → 继续查询 → 综合判断 → 输出结论
```

### 典型问题

用户问「为什么产品销量下降了？」——没有固定资料入口，可能涉及：
1. 查询销售数据库确认下降趋势
2. 查询价格历史看是否有调价
3. 查询营销活动记录看投放是否减少
4. 查询客服工单看是否有集中投诉
5. 查询库存系统看是否有缺货
6. 证据不足时继续选择新工具或数据源

### Agent 可调用的工具

内部文档搜索、向量数据库、知识图谱查询、SQL 数据库、Web API、工单系统、日志系统、BI 报表……

### 适用场景

- 复杂问题分析、业务异常归因
- 跨系统调查、多数据源综合判断
- 技术排障、竞品分析、投资研究、运营复盘

### 代价

- **延迟高**：多轮工具调用，不可预测
- **成本高**：每次规划/判断/重规划都消耗 token
- **调试难**：不止看检索结果，还要看 Agent 为什么选这个工具、为什么跳过那个数据源

> Agentic RAG 不是默认更高级，只有真正需要它的场景才上——如果只是 FAQ，用 Agentic RAG 就像为拧一颗螺丝搬来一台挖掘机。

> Agentic RAG 解决的是「**面对复杂问题，下一步该怎么行动**」的问题。

---

## 三种架构对比

| 维度 | Classic RAG | Graph RAG | Agentic RAG |
|------|------------|-----------|-------------|
| **核心动作** | 检索 | 连接 | 推理 |
| **解决什么问题** | 资料在哪里 | 资料之间怎么连 | 下一步该查什么 |
| **适合问题** | 一跳问答、文档查询、FAQ | 依赖分析、影响分析、供应链 | 多步骤调查、跨系统分析 |
| **主要优势** | 快、简单、成本低 | 能沿关系路径查找和解释 | 灵活，能动态选择工具 |
| **典型代价** | 不擅长关系链和多跳 | 建图与维护成本高 | 延迟高、成本高、调试难 |
| **查询路径** | 固定 | 沿图谱遍历 | 动态规划 |

---

## 如何选择：看问题形状，不看架构名字

### 第一问：问题能否一步命中答案？
如果答案可以直接从某个文档片段中找到 → **Classic RAG**（产品保修期、政策适用范围、功能配置方式）

### 第二问：答案是否依赖关系链？
如果答案需要跨实体、跨文档、跨关系查找 → 加入 **Graph RAG**（谁影响谁、谁依赖谁、谁和谁有关）

### 第三问：查询路径能否提前定义？
如果一开始就不知道该查哪个系统、哪类数据 → 考虑 **Agentic RAG**（销量下降归因、故障排查、综合研究）

### 工程实践原则

**先用简单方案解决 80% 的问题，再为真正复杂的问题增加复杂度。**

```
1. 先上 Classic RAG → 解决高频、明确、低成本的问题
2. 遇到关系链问题 → 引入 Graph RAG
3. 遇到开放式多步骤跨系统调查 → 交给 Agentic RAG
```

架构的价值不在于名字，而在于它能不能匹配你的问题。

## 从 RAG 到 LLM Wiki：知识库技术演进

[[AI 知识库技术演进拆解：从 RAG 到 NotebookLM，再到 LLM Wiki]] 指出，AI 知识库的技术路径经历三个阶段：

1. **第一阶段：低配 RAG** — 资料切块 → 向量化 → 检索 → 回答
2. **第二阶段：NotebookLM 类产品化 RAG** — 文档理解 → 多索引 → Retrieval and Ranking → Context Engineering → Source Grounding
3. **第三阶段：LLM Wiki / 深度知识库** — 知识抽取 → 实体识别 → 主题页生成 → 关系链接 → 冲突检测 → 增量更新 → 持续演化

三个阶段是逐层叠加的过程：低配 RAG 是底座，NotebookLM 把底座产品化/自动化/可信化，LLM Wiki 再把知识结构沉淀为长期资产。详见 [[obsidian-llm-wiki-practice|Obsidian LLM Wiki 实践]]。
