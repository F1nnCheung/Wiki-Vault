# RAG 评估体系

RAG 系统的评估不能靠"感觉"——需要一套量化指标体系，将检索质量与生成质量分开度量。核心思路：**检索层看"找没找到"，生成层看"说没说对"**，两层问题独立诊断才能精准定位瓶颈。

## 为什么评估必须分层

RAG 系统的链路是：Query → 检索层（Retriever）→ 召回文档块 → 生成层（Generator）→ 最终 Answer。这两层的问题是独立的：

- **检索层问题**：召回的文档和 Query 不相关；或者相关文档被排在第 10 名，前 3 名全是废文
- **生成层问题**：相关文档召回了，但 LLM 没用到（幻觉）；或者答案风马牛不相及

如果只用 LLM 打分看"最终答案好不好"，你无法知道是哪个环节出了问题——就像一道菜不好吃，你不知道是食材不行还是厨师手艺差。

## RAG 失败模式

在讨论评估指标之前，先明确 RAG 系统会在哪里失败。

### 检索层失败

| 失败模式 | 描述 |
|----------|------|
| **召回缺失** | 相关文档存在于知识库，但没有被检索出来（嵌入模型语义捕捉能力不足、分块策略不当） |
| **检索噪声** | Top-K 文档中混入大量不相关内容，稀释了有用上下文 |
| **语义漂移** | 查询与文档的向量表示存在分布偏差，相似度计算失去参考意义 |
| **知识过期** | 索引中的内容是陈旧的，检索到的"事实"已不再准确 |

### 生成层失败

| 失败模式 | 描述 |
|----------|------|
| **忠实度缺失（幻觉）** | 模型无视检索内容，用自身参数知识生成答案 |
| **选择性忽略** | 检索内容存在，模型却只采用其中一部分，遗漏关键信息 |
| **过度依赖上下文** | 检索内容本身有误，模型忠实复述了错误信息（最危险也最难发现） |
| **答案偏题** | 技术上忠实于上下文，但答案没有真正回应用户的问题 |

## 核心评估指标体系

RAG 评估指标分为三层：**检索指标 → 生成指标 → 端到端指标**。

### 检索层指标

检索指标衡量"系统能否找到正确的文档"。

#### Recall@K（召回率@K）

最基础的检索指标。回答：对于一个 Query，所有相关文档中，有多少比例出现在召回的前 K 条里？

```
Recall@K = 前K条里的相关文档数 / 总相关文档数
```

- **盲区**：只管"有没有被召回"，不管排序。第 1 名和第 10 名对它来说没区别。但在 RAG 里通常只把 Top3 塞给 LLM——如果相关文档都在 Top8-10，Recall@10=1.0 但实际没用。

#### MRR（Mean Reciprocal Rank，平均倒数排名）

关注**第一个**相关文档出现的位置，衡量系统"把最重要的东西排在前面"的能力。

```
MRR = (1/|Q|) × Σ (1 / rank_i)
```

- 第 1 名 → 1.0，第 2 名 → 0.5，第 5 名 → 0.2，第 10 名 → 0.1
- 排名越靠后，得分衰减越快
- **盲区**：只看第一个相关结果，如果一个 Query 有多个相关文档，后续排名的好坏完全不管
- **适用场景**：每个 Query 只有一个最正确答案时（如 FAQ 检索）

#### NDCG（Normalized Discounted Cumulative Gain）

最全面的检索排序指标。同时考虑：
- 每个文档的相关性**程度**（可分 0/1/2 等级，而非简单的相关/不相关）
- 文档所在**位置**（越靠前权重越高）

```
DCG@K = Σ (rel_i / log₂(i+1))
NDCG@K = DCG@K / IDCG@K（实际值 / 理想排序值）
```

- NDCG 永远在 [0, 1] 之间
- 与 MRR 不同：NDCG 能区分"两个相关文档排第1和第2"vs"排第1和第5"

#### Precision@K（精确率@K）

在检索出的前 K 个文档中，真正相关文档的比例。高 Precision@K 意味着检索结果"含金量高"，噪声少。

### 生成层指标（RAGAS 四大指标）

生成指标衡量"模型基于检索内容生成的答案质量"。

#### Faithfulness（忠实度）

答案中的每个声明，是否都能在检索文档中找到支撑？这是衡量**幻觉**最直接的指标。

```
忠实度 = 有检索文档支撑的声明数 / 答案中的总声明数
```

实现方式：将答案分解为若干原子声明，逐一用 LLM 评判者（LLM-as-a-Judge）核查是否有文档依据。

#### Answer Relevancy（答案相关性）

答案是否真正回答了用户的问题？答案可以完全忠实于检索内容，却并不相关——例如问"价格"，答案却是"功能介绍"。

#### Context Precision（上下文精确率）

检索到的文档中，真正对回答有帮助的比例。精确率低意味着大量无关文档进入上下文，会稀释注意力或引入干扰。

#### Context Recall（上下文召回率）

生成正确答案所需的信息，在检索结果中的覆盖程度。召回率低意味着关键信息被遗漏，模型被迫"猜测"或"幻觉"。

### 端到端指标

| 指标 | 描述 |
|------|------|
| **事实正确性** | 答案在现实世界中是否正确？与忠实度的区别：忠实度关注"与检索内容是否一致"，事实正确性关注"与真实世界是否一致"。如果检索内容本身有误，忠实度可以很高但事实正确性很低 |
| **幻觉率** | 答案中无法从检索内容或真实知识中找到依据的声明比例 |
| **延迟（Latency）** | 端到端响应时间。大多数实时应用目标 3 秒以内。延迟最大占比通常在 LLM 生成环节 |
| **成本（Cost per Query）** | 每次请求消耗的 Token 数。高并发生产场景下，成本是与准确性同等重要的指标 |

## 构建评估测试集

评估指标需要好的测试数据。测试集的质量直接决定评估结论的可信度。

### 黄金测试集（Golden Dataset）

由领域专家人工构建并验证的问答对，包含：问题、参考答案、对应正确上下文来源。

**关键原则**：每个评估周期都要冻结测试集版本。若测试集随时间变化，跨周期的指标对比就失去意义。

### 合成测试集

当黄金数据稀缺时，可用 RAGAS、ARES 等工具自动生成合成问题。**注意**：合成数据需要人工审核验证，防止模型学习到合成数据的独有模式导致评估虚高。

### 测试集建议构成

| 类型 | 说明 | 占比建议 |
|------|------|----------|
| 简单事实问题 | 直接可从单一文档回答 | 30% |
| 多跳推理问题 | 需跨多个文档综合推理 | 25% |
| 边界场景问题 | 知识库中无答案的问题 | 20% |
| 对抗性问题 | 实体替换、语义相近但答案不同 | 15% |
| 歧义问题 | 多意图或表述模糊的查询 | 10% |

### 对抗性测试集

专门构造"混淆的错误用例"（硬负例），与正确答案在语义上非常接近，但在关键事实上存在错误（如将人名、日期、数字替换）。很多主流评估框架无法区分"实体正确但答案错误"的文档和"真正正确"的文档，对抗性测试集是检验评估系统真实能力最有效的方式。

## 工业级评估三大流程

将评估体系落地到生产环境，需要三条并行的流程管线。

### 流程一：离线评估（Offline Evaluation）

在实验环境中，用带标注的测试集运行自动化评估：

1. 准备版本化的黄金测试集或合成测试集
2. 对每个 query 执行完整 RAG 流程
3. 计算检索层指标（Recall@K、MRR、NDCG）和生成层指标（Faithfulness、Answer Relevancy 等）
4. 对比不同配置/版本的指标变化

### 流程二：在线评估（Online Evaluation）

在生产流量中匿名采集用户反馈：

- **显式反馈**：👍/👎 按钮、用户打分
- **隐式反馈**：答案复制率、阅读时长、追问频率、是否重新生成
- **A/B 测试**：新版本检索策略 vs 旧版本，对比核心指标

### 流程三：持续监控（Continuous Monitoring）

生产环境中的实时监控面板：

- **检索质量漂移**：嵌入模型迭代或知识库更新导致 Recall 缓慢下滑
- **生成质量异常**：Faithfulness 突然下降（可能是 prompt 变更或模型切换）
- **成本异常**：单次查询 Token 消耗突增
- **延迟告警**：P95/P99 延迟超过阈值

## 选型建议

| 场景 | 推荐指标组合 | 评估工具 |
|------|-------------|----------|
| 快速原型验证 | Recall@5 + 人工抽查 | 手动 |
| 检索策略选型 | Recall@K + MRR + NDCG | Ragas / LlamaIndex Evaluation |
| 生产质量把关 | Faithfulness + Answer Relevancy + Context Recall | Ragas + 人工标注黄金集 |
| 持续迭代优化 | 全套检索+生成指标 + 在线 A/B | Ragas + LangSmith / MLflow |

## 总结

- RAG 评估的核心是**分层诊断**：检索层和生成层必须分开评估，否则无法定位瓶颈
- 检索层核心三指标：Recall@K（召回覆盖）、MRR（首个相关排位）、NDCG（排序质量）
- 生成层核心四指标：Faithfulness（反幻觉）、Answer Relevancy（答即所问）、Context Precision（上下文纯度）、Context Recall（上下文覆盖）
- 测试集质量决定评估可信度：黄金集 + 对抗样本 + 版本冻结
- 工业级落地需要离线评估 + 在线评估 + 持续监控三条管线并行
