# 知识图谱（Knowledge Graph）

> 知识图谱不是"画一张漂亮的图"，而是**把散落在表格、文档、网页、系统和人脑里的知识关系，变成计算机能理解和复用的结构**。它用"实体"和"关系"来组织知识——当问题的关键不在单条数据，而在对象之间的关系时，知识图谱开始有价值。

## 一句话定义

**用"实体"（节点）和"关系"（边）来组织知识的图结构。**

三个关键要素：

| 要素 | 含义 | 示例 |
|------|------|------|
| **实体** | 可识别的对象 | 乔布斯、苹果公司、iPhone |
| **关系** | 实体之间的连接 | 创立、发布、属于、位于 |
| **图结构** | 节点（实体）+ 边（关系）= 可查询的关系网 | 乔布斯→创立→苹果公司→发布→iPhone |

---

## 知识的基本形式：三元组

```
乔布斯 → 创立 → 苹果公司
苹果公司 → 发布 → iPhone
iPhone → 属于 → 智能手机
北京 → 是首都 → 中国
```

一条三元组记录一个事实。众多三元组连接起来，就构成了知识图谱。

---

## 为什么不能只用数据库？

关系型数据库优化的是"把数据按表存好"，而知识图谱优化的是"**把关系表达清楚**"。

### 金融风控案例

单独看任何一张表，每个用户看起来都正常。但把关系连起来会看到隐藏的风险：

```
用户A → 使用 → 设备X ← 使用 ← 用户B → 使用 → 设备X ← 使用 ← 用户C
用户A → 绑定 → 银行卡Y ← 收款至 ← 用户B
用户A、B、C → 登录自 → 同一IP地址
```

**风险不在某一条数据里，而在关系构成的模式中。** 数据库可以通过 JOIN 查询关系，但当关系类型多、跳转层数深、数据来源杂时，表结构会变得复杂难维护。

> 知识图谱不是替代数据库，而是在数据库和应用之间增加一层"**关系理解能力**"。

---

## 与搜索、向量库的差异

```
数据库      → 擅长存储精确的记录
搜索引擎    → 擅长找包含关键词的文本
向量数据库  → 擅长找语义相似的内容
知识图谱    → 擅长表达实体之间的关系
```

用户问"上次 Chat 接口为什么重构？"时：
- **关键词搜索**：找到包含这些词的文档
- **向量检索**：找到语义相近的段落
- **知识图谱**：不仅找到文档，还能展现 Chat 接口属于哪个系统、依赖哪些服务、历史上出现过什么错误、谁负责过这条链路、重构影响了哪些上下游

## 核心概念

### 实体：身份很关键

难点不在识别一个词，而在**确认它到底指向哪个对象**。
- "苹果" → 水果？苹果公司？内部项目代号？
- "张伟" → 销售同事？客户联系人？

**实体识别和实体消歧**是图谱的第一项关键任务。

### 关系：不是所有连线都有意义

一条有意义的关系必须能回答：它表示什么语义？有方向吗？有时间范围吗？来源是什么？能被业务解释吗？

```
员工A → 负责 → 项目B → 依赖 → 系统C → 出现 → 错误D → 记录于 → 日志E
```

这条路径能清楚地解释"为什么项目B受影响"。

### 本体（Ontology）：给图谱定规则

本体是知识图谱的架构设计图，规定：
- 有哪些实体类型？有哪些关系类型？
- 哪些关系是合法的？（"人"可以有"出生地"，"公司"不能有）
- 实体之间的层级关系是什么？

没有本体约束，图谱容易变成一堆杂乱的连线。

### 来源和置信度：可追溯很重要

```
来自官网 → 高信任
来自数据库 → 中高信任
来自模型自动抽取 → 低信任
来自人工审核 → 最高信任
```

在金融、医疗、法律场景，必须能说清楚每条关系从哪来、什么时候产生、谁审核过、还有效吗。

## 构建流程

```
确定业务问题 → 设计 Schema/本体 → 接入数据源 → 抽取实体和关系
→ 实体链接与消歧 → 融合多源数据 → 质量校验 → 存储查询 → 接入应用 → 持续迭代
```

**关键原则**：第一版不追求大而全。把核心实体和核心关系连准，比堆砌很多类型更重要。

数据源分三类：
- **结构化**（数据库、表格）→ 规则直接映射
- **半结构化**（网页、JSON）→ 解析和转换
- **非结构化**（文档、邮件、工单）→ NLP/大模型抽取（需经过验证才能入库）

---

## 应用场景

| 场景 | 核心价值 |
|------|---------|
| **搜索** | 从"字符串匹配"走向"实体理解"（Google: "Things, not strings"） |
| **推荐系统** | 不只"买了A也买了B"，能理解商品类目、品牌关系、场景偏好 |
| **金融风控** | 发现隐藏在多跳关系里的团伙欺诈模式 |
| **医疗医药** | 疾病-症状-药物-基因-指南之间的关系天然适合图谱 |
| **企业知识管理** | 把项目、系统、接口、人员、文档、决策和故障联系起来 |
| **GraphRAG** | 沿图谱找上下文 → 交给大模型生成答案，给 AI 提供更稳固的事实骨架 |

---

## 知识图谱 + 大模型 = GraphRAG

传统 RAG：`文档切块 → 向量检索 → 召回文本 → 大模型生成答案`

GraphRAG：`识别实体和关系 → 构建图结构 → 沿着图谱找相关上下文 → 交给大模型生成答案`

知识图谱不替代大模型，而是**给大模型提供更稳固的事实骨架**。特别适合：
- 跨多文档的问题
- 多实体、多跳关系查询
- 需要解释来源和路径的答案

---

## 核心洞见

> 知识图谱的真正价值不在于把数据画成一张网，而在于**把原本散落、隐含、难以复用的关系显式化**。

> 判断一个场景是否需要知识图谱：**当问题的关键不在单条数据，而在对象之间的关系；当答案不仅要给结果，还要能解释来源和路径**——知识图谱就开始有价值。
